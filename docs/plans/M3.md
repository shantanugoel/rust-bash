# Milestone 3: Execution Safety — Implementation Plan

## Problem Statement

rust-bash is designed to run untrusted, AI-generated scripts. M1 and M2 delivered a correct interpreter with ~60 commands, but execution safety is only partially enforced. Of the 10 defined `ExecutionLimits`, 3 are not enforced at all (`max_string_length`, `max_substitution_depth`, `max_heredoc_size`), and several enforced limits use inconsistent error types (`Execution` or `Timeout` instead of `LimitExceeded`). Network access (curl) is not yet implemented. M3 closes these gaps to deliver a fully bounded, policy-controlled execution environment.

## M1/M2 Carryover

### Current Limit Enforcement Status

| Limit | Enforced? | Error Type | Issue |
|-------|-----------|-----------|-------|
| `max_call_depth` | ✅ | `Execution` | Should be `LimitExceeded` |
| `max_command_count` | ✅ | `LimitExceeded` | OK |
| `max_loop_iterations` | ✅ | `Execution` | Should be `LimitExceeded` |
| `max_execution_time` | ✅ | `Timeout` | OK — `Timeout` is semantically correct |
| `max_output_size` | ✅ | `LimitExceeded` | OK |
| `max_glob_results` | ✅ | Silent cap | Should error with `LimitExceeded` |
| `max_brace_expansion` | ✅ | `LimitExceeded` | OK |
| `max_string_length` | ❌ | — | Not enforced anywhere |
| `max_substitution_depth` | ❌ | — | Not enforced anywhere |
| `max_heredoc_size` | ❌ | — | Not enforced anywhere |

### Other Issues to Address

1. **`RustBashError::LimitExceeded` is not structured.** Currently `LimitExceeded(String)`, but Ch. 7 specifies structured fields: `{ limit_name, limit_value, actual_value }`.
2. **`RustBashError::Network` variant is missing.** Ch. 2 lists it in the error hierarchy, but `error.rs` doesn't define it.
3. **Command substitution resets counters.** `execute_command_substitution()` (expansion.rs:213) creates `ExecutionCounters::default()`, meaning nested `$(...)` gets a fresh command budget. This should share the parent's counters (at least for `command_count` and `start_time`) to prevent circumventing limits. Same issue in `execute_subshell()` (walker.rs:695) and `make_exec_callback()` (walker.rs:826).
4. **`source` and `eval` don't increment `call_depth`.** Both builtins call `execute_program()` on the parent state without tracking depth. A chain of `source` or `eval 'eval ...'` calls can blow the stack without hitting `max_call_depth`.

## Approach

Split into two phases matching the guidebook's M3.1 (limits) and M3.2 (network). Limits first because they're foundational; network depends on having a working curl command with policy enforcement.

### Dependencies

**New crates needed:**
- `ureq` — minimal blocking HTTP client for curl implementation (latest stable). Chosen over `reqwest` to avoid async runtime dependency and keep binary size small. No feature-gating initially; add later only if binary size becomes a measured problem.
- `url` — URL parsing for prefix validation in `NetworkPolicy`. Needed in Phase 3 (not Phase 4).

## Phase 1 — Error Type Refactoring (Foundation)

> Make the error types match the spec before adding new enforcement points.

### 1a. Restructure `RustBashError::LimitExceeded`

Change from `LimitExceeded(String)` to a structured variant:

```rust
RustBashError::LimitExceeded {
    limit_name: &'static str,
    limit_value: usize,
    actual_value: usize,
}
```

Using `&'static str` for `limit_name` avoids allocation on error paths and enables clean pattern matching in tests (e.g., `LimitExceeded { limit_name: "max_command_count", .. }`).

Update all existing `LimitExceeded` construction sites:
- `walker.rs:check_limits()` — `max_command_count`, `max_output_size`
- `brace.rs:check_limit()` — `max_brace_expansion`

Update the `Display` impl and any pattern matches on `LimitExceeded` across the codebase. Known sites requiring update:
- `tests/integration.rs` — at least 2 tests that match on `LimitExceeded(msg)` and assert `msg.contains(...)`
- Any other `match` arms on `RustBashError`

### 1b. Add `RustBashError::Network` variant

```rust
RustBashError::Network(String)
```

Add to `Display` impl, `std::error::Error` impl. Not yet used — just plumbing for Phase 4.

### 1c. Migrate existing limit errors to correct variant

Change these callsites from `RustBashError::Execution` to `RustBashError::LimitExceeded`:
- `walker.rs:execute_function_call()` — `max_call_depth`
- `walker.rs:execute_for()` — `max_loop_iterations`
- `walker.rs:execute_arithmetic_for()` — `max_loop_iterations`
- `walker.rs:execute_while_until()` — `max_loop_iterations`

### 1d. Change glob limit from silent cap to error

In `expansion.rs:glob_expand_words()`, change from silently taking the first N results to returning `RustBashError::LimitExceeded` when glob results exceed `max_glob_results`. This requires changing the function signature from returning `Vec<String>` to `Result<Vec<String>, RustBashError>` and propagating the error through `expand_word()` / `expand_word_mut()`.

**Exit criteria**: All existing limits produce structured `LimitExceeded` errors. All existing tests still pass (update test assertions as needed). `cargo clippy -- -D warnings` clean.

## Phase 2 — Enforce Unenforced Limits (M3.1 core)

### 2a. `max_string_length` enforcement

Check at these specific chokepoints:

1. **`set_variable()`** (interpreter/mod.rs) — catches all direct variable assignments:
   ```rust
   if value.len() > state.limits.max_string_length {
       return Err(RustBashError::LimitExceeded { ... });
   }
   ```

2. **`expand_word_to_string_mut()`** (expansion.rs) — single chokepoint for all assignment-like contexts (heredocs, redirects, case values, `${var:-default}` etc.)

3. **`builtin_read()`** (builtins.rs) — the value read from stdin before assignment

For inline usage like `echo "$huge"`, the `max_output_size` limit provides the backstop, so we don't need to check every expansion intermediate.

### 2b. `max_substitution_depth` enforcement

Add a `substitution_depth` field to `ExecutionCounters`:

```rust
pub substitution_depth: usize,  // init: 0
```

Update the manual `Default` impl for `ExecutionCounters` (mod.rs) and the `reset()` method to include `substitution_depth: 0`.

In `execute_command_substitution()` (expansion.rs), increment before recursing and check:

```rust
state.counters.substitution_depth += 1;
if state.counters.substitution_depth > state.limits.max_substitution_depth {
    state.counters.substitution_depth -= 1;
    return Err(RustBashError::LimitExceeded { ... });
}
// ... execute ...
state.counters.substitution_depth -= 1;
```

This catches `$($($($(...)))) ` nesting bombs. The depth counter must be passed into the sub_state rather than resetting it.

### 2c. `max_heredoc_size` enforcement

In `walker.rs` where heredocs are processed (the `HereDocument` arm at line ~1004), check the body size after expansion:

```rust
ast::IoRedirect::HereDocument(fd, heredoc) => {
    let body = ...;  // existing expansion
    if body.len() > state.limits.max_heredoc_size {
        return Err(RustBashError::LimitExceeded { ... });
    }
    ...
}
```

Also check the `HereString` arm (line ~997) for the same size limit — a `<<<$(generate_huge_string)` could produce arbitrarily large stdin. Reuse `max_heredoc_size` for both.

### 2d. Fix counter sharing across subshells, command substitutions, and exec callbacks

Currently, `execute_command_substitution()` (expansion.rs:213), `execute_subshell()` (walker.rs:695), and `make_exec_callback()` (walker.rs:826) all reset counters with `ExecutionCounters::default()`. This means nested execution gets a fresh command budget and a fresh `start_time`, completely defeating the parent's limits.

Fix: propagate key counters into sub-states:
- `command_count` → share (add to parent's total after sub-state returns)
- `output_size` → share (same)
- `start_time` → share (copy from parent, so wall-clock is global)
- `call_depth` → keep at 0 (function depth is per-scope)
- `substitution_depth` → propagate (for nested `$()`)

For `execute_command_substitution()` and `execute_subshell()`: initialize sub-state counters from parent, fold `command_count` and `output_size` back into the parent after completion.

For `make_exec_callback()`: the closure signature `Fn(&str) -> Result<CommandResult, RustBashError>` doesn't carry counter state back. At minimum, capture the parent's `start_time` (it's `Copy`) so wall-clock limits apply globally. For `command_count`, accept this as a known limitation — the parent's `dispatch_command` still counts the `xargs`/`find` call itself. Document this trade-off.

### 2e. Enforce `call_depth` for `source` and `eval`

In `builtins.rs`, both the `source` builtin (line ~813) and the `eval` builtin (line ~705) call `execute_program()` on the parent state without tracking depth. Add increment/decrement of `counters.call_depth` around the `execute_program()` call in both, similar to `execute_function_call()`. This prevents deeply nested `source`/`eval` chains from overflowing the stack.

**Exit criteria**: All 10 limits enforced. Unit tests for each enforcement point. All existing 354+ tests pass.

## Phase 3 — `NetworkPolicy` Type & Builder Integration

### 3a. Define `NetworkPolicy` struct

Add `url = "2"` to `Cargo.toml` (needed for URL normalization in `validate_url()`).

Create `src/network.rs`:

```rust
use std::collections::HashSet;
use std::time::Duration;

pub struct NetworkPolicy {
    pub enabled: bool,                     // default: false
    pub allowed_url_prefixes: Vec<String>,
    pub allowed_methods: HashSet<String>,  // default: {"GET", "POST"}
    pub max_redirects: usize,             // default: 5
    pub max_response_size: usize,         // default: 10MB
    pub timeout: Duration,                // default: 30s
}
```

With a `Default` impl that has `enabled: false`. Derive `Clone` (needed for `make_exec_callback` closure capture).

Add a `validate_url(&self, url: &str) -> Result<(), String>` method that checks the URL against `allowed_url_prefixes` using literal prefix matching (as specified in Ch. 7):
- Parse URL with `url::Url` to normalize
- Compare against each prefix
- Reject if no prefix matches

Add a `validate_method(&self, method: &str) -> Result<(), String>` method.

### 3b. Wire into `InterpreterState` and builder

- Add `network_policy: NetworkPolicy` field to `InterpreterState`
- Add `network_policy: Option<NetworkPolicy>` field to `RustBashBuilder`
- Add `.network_policy(policy)` method to `RustBashBuilder`
- Pass through in `build()`, defaulting to `NetworkPolicy::default()` (disabled)
- Add `NetworkPolicy` to the public re-exports in `lib.rs`

### 3c. Pass `NetworkPolicy` to commands

Add `network_policy: &NetworkPolicy` field to `CommandContext`. All commands receive it; only `curl` will use it.

**Cascade note**: `CommandContext` is constructed in `dispatch_command()` (walker.rs) and `make_exec_callback()` (walker.rs). The exec callback closure must also capture `network_policy` (it's `Clone`). Additionally, ~10 test helper `ctx()` functions across command modules (`utils.rs`, `text.rs`, `file_ops.rs`, `navigation.rs`, `exec_cmds.rs`, `test_cmd.rs`, `diff_cmd.rs`, `sed.rs`, `jq_cmd.rs`, `awk/mod.rs`) must include the new field. The compiler will catch all missing sites.

**Exit criteria**: `NetworkPolicy` compiles, is configurable via the builder, and is threaded into `CommandContext`. No behavioral change yet. Phase 3c must be sequenced after Phase 2 completes (both touch `dispatch_command` and `make_exec_callback`).

## Phase 4 — `curl` Command (M3.2)

### 4a. Add `ureq` dependency

```toml
ureq = "3"
```

Use the latest stable version. Note: `url` was already added in Phase 3a. The ureq 3.x API is agent-based (no top-level `ureq::get()`); implementation must target the v3 API.

### 4b. Implement `CurlCommand`

Create `src/commands/net.rs` with a `CurlCommand` implementing `VirtualCommand`.

**Supported flags** (practical subset for AI agent use):
- URL argument (positional or via explicit flag)
- `-X METHOD` / `--request METHOD` — HTTP method (default: GET)
- `-H "Header: Value"` / `--header` — custom headers (repeatable)
- `-d DATA` / `--data` — request body (switches default method to POST)
- `-o FILE` / `--output` — write response to VFS file instead of stdout
- `-s` / `--silent` — suppress progress (no-op since we don't show progress, but accept the flag)
- `-S` / `--show-error` — show errors even with `-s` (accept, no-op)
- `-f` / `--fail` — return exit code 22 on HTTP errors (4xx/5xx)
- `-L` / `--location` — follow redirects (up to `max_redirects`, each target re-validated)
- `-i` / `--include` — include response headers in output
- `-w FORMAT` / `--write-out` — output format string after transfer (support `%{http_code}`)
- `-I` / `--head` — HEAD request only
- `-v` / `--verbose` — print request/response details to stderr
- `-k` / `--insecure` — accept flag but ignore (we always verify TLS)

### 4c. Network policy enforcement in `CurlCommand`

Before making any HTTP call:
1. Check `network_policy.enabled` — if false, return error: `"curl: network access is disabled"`
2. Validate URL against `allowed_url_prefixes` — reject with: `"curl: URL not allowed by network policy: {url}"`
3. Validate HTTP method against `allowed_methods`
4. Set request timeout from `network_policy.timeout`
5. Limit response body size to `network_policy.max_response_size` (read in chunks, abort if exceeded)
6. For `-L` (follow redirects): validate each redirect target URL, enforce `max_redirects`

### 4d. Register `CurlCommand`

Add to `commands::register_default_commands()` in `commands/mod.rs`.

**Exit criteria**: `curl` works against allowed URLs when network is enabled, returns clear errors otherwise. Redirect chains are validated at each hop. Response size is bounded.

## Phase 5 — Tests & Documentation

### 5a. Limit enforcement tests

For each limit, write integration tests that:
1. Set a low limit via the builder
2. Run a script that exceeds it
3. Assert a `LimitExceeded` error with the correct `limit_name`
4. Verify the shell remains usable after (subsequent `exec()` succeeds)

Specific test cases:
- `max_command_count`: `for i in $(seq 1 20); do echo $i; done` with limit 10
- `max_loop_iterations`: `while true; do :; done` with limit 100
- `max_call_depth`: recursive function `f() { f; }; f` with limit 5
- `max_execution_time`: `sleep 999` with 100ms timeout (sleep caps to max_execution_time, so this runs ~100ms)
- `max_output_size`: `yes | head -n 100000` with 1KB limit
- `max_string_length`: `x=""; for i in $(seq 1 1000); do x="${x}aaaa"; done` with 1KB limit
- `max_substitution_depth`: `echo $(echo $(echo $(echo x)))` with depth 2
- `max_heredoc_size`: heredoc with large body, limit 100 bytes
- `max_glob_results`: create many files, glob `*` with limit 5
- `max_brace_expansion`: `echo {1..10000}` with limit 100
- Counter sharing: `for i in 1 2 3; do echo $(seq 1 100); done` with `max_command_count` 200 — should fail because subshell commands accumulate
- `source` call depth: nested source files hitting `max_call_depth`
- `eval` recursion depth: `eval 'eval "eval \"echo done\""'` with `max_call_depth: 2`
- Counter reset between exec() calls: `shell.exec("heavy")` then `shell.exec("echo ok")` succeeds
- `max_string_length` in `read`: `echo "$huge" | read var` where input exceeds limit
- Here-string size: `cat <<<$(generate_huge)` with low `max_heredoc_size`

### 5b. Network policy tests

All HTTP tests should be policy-validation unit tests — **do not make integration tests that hit real URLs**. Add `#[cfg(feature = "network-integration-tests")]` gate if real HTTP tests are ever desired.

- Network disabled (default): `curl https://example.com` → error
- Network enabled, URL allowed: unit-test that `validate_url()` returns Ok
- Network enabled, URL rejected: wrong prefix → error
- URL normalization attack: `https://api.example.com@evil.com/` must be rejected (parsed by `url::Url`, the prefix check on the normalized URL catches this)
- Redirect validation: allowed URL redirects to disallowed URL → error
- Response size limit exceeded → error
- Method restriction: POST when only GET allowed → error

### 5c. Documentation updates

- Update the threat matrix in `docs/guidebook/07-execution-safety.md`: change **(planned)** to ✅ for all limits enforced
- Update `docs/guidebook/10-implementation-plan.md`: mark M3.1 and M3.2 with ✅
- Update `README.md` if the builder API changed (e.g., `network_policy` method)

**Exit criteria**: All tests pass. `cargo fmt && cargo clippy -- -D warnings && cargo test` clean. Documentation matches implementation.

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `ureq` adds too much binary bloat | Low | Medium | ureq is minimal (~200KB). Feature-gate `curl` behind a cargo feature only if measured as a problem |
| Counter-sharing in subshells changes semantics | Medium | Medium | Only share `command_count`, `output_size`, `start_time`. Keep `call_depth` per-scope |
| Glob error (Phase 1d) breaks existing scripts | Medium | Low | Only error when exceeding limit; default is 100K which is very high |
| `max_string_length` enforcement catches legitimate large outputs | Low | Medium | Default is 10MB — very generous. Users can override |
| Test flakiness for `max_execution_time` | Medium | Low | Use generous margins (e.g., 100ms limit for sleep 999) |
| `InterpreterState` construction cascade | Low | Low | ~5 sites must add `network_policy` field; compiler catches all |
| `ureq` v3 API differs from v2 | Low | Low | Target v3 agent-based API from the start |

## Resolved Design Decisions

1. **`sleep` command and timeout interaction**: The existing `sleep` implementation already caps duration to `max_execution_time` (utils.rs:321-325). A `sleep 999` with a 5s limit blocks for ~5s, then the next `check_limits()` catches the timeout. This is acceptable — the capping ensures it's bounded. No change needed.

2. **Feature-gate curl?** No. Binary size impact of ureq is minimal. Feature flags add CI complexity disproportionate to the benefit. Revisit only if binary size becomes a measured problem.

3. **Exec callback counter limitation**: The `make_exec_callback` closure cannot fold counters back (signature is `Fn(&str) -> Result<CommandResult, RustBashError>`). We propagate `start_time` for wall-clock enforcement and accept that per-invocation command counts reset. The parent still counts the top-level `xargs`/`find` invocation. Document as known limitation.
