# Milestone 6: Shell Language Completeness â€” Implementation Plan

## Problem Statement

rust-bash has a correct interpreter (M1), text-processing commands (M2), execution safety (M3), filesystem backends (M4), and integration targets (M5). However, real-world AI-generated bash scripts frequently use language features that are currently parsed but not executed: arrays, shopt options, advanced builtins (`mapfile`, `getopts`, `type`, `command`), process substitution, parameter transformations (`${var@Q}`, `${!ref}`), and special variables (`$PIPESTATUS`, `$FUNCNAME`, `$SECONDS`). Many `set` options (`-x`, `-a`, `-f`, `-n`, `noclobber`) are accepted syntactically but produce no behavioral effect.

M6 closes these gaps so that bash scripts using arrays, shopt, advanced builtins, and process substitution work without modification. M6.12 (differential testing) was planned and partially implemented separately â€” see `docs/plans/M6.12.md`.

## Prerequisites & Current State

**M1â€“M5 are complete.** The interpreter handles:
- Simple/compound commands, pipelines, functions, loops, case, arithmetic, globs, brace expansion, here-docs
- 17 builtins: `exit`, `cd`, `export`, `unset`, `set`, `shift`, `readonly`, `declare`, `read`, `eval`, `source`, `break`, `continue`, `:`, `let`, `local`, `return`, `trap`
- `Variable` is scalar-only: `{ value: String, exported: bool, readonly: bool }`
- `ShellOpts` tracks 4 flags: `errexit`, `nounset`, `pipefail`, `xtrace`
- `declare` supports only `-r` (readonly) and `-x` (export)
- `read` supports only `-r` (raw) and `-p` (prompt, no-op)
- `printf` is an external command (not a builtin) with standard format specifiers but no `-v`, `%b`, or `%q`

**Parser support exists but is discarded for:** `AssignmentValue::Array`, `ArrayElementName`, `Parameter::NamedWithIndex`, `Parameter::NamedWithAllIndices`, `ParameterExpr::MemberKeys`, `ProcessSubstitution`, and all `ParameterTransformOp` variants (currently return value unchanged or empty string).

**BASH_REMATCH** is stored as flat variables (`BASH_REMATCH`, `BASH_REMATCH_1`, `BASH_REMATCH_2`, etc.) rather than as an indexed array. **PIPESTATUS** is not exposed â€” `exit_codes` is collected in `execute_pipeline` but discarded. **$LINENO** returns hardcoded `"0"`.

## Key Design Challenges

### 1. Variable Type System Evolution

The `Variable` struct must grow from scalar-only to support indexed arrays, associative arrays, and attribute flags â€” without bloating the common case (most variables are plain strings). Every call site that reads or writes `Variable.value` must be audited.

**Solution**: Replace the flat fields with an enum + bitflags:

```rust
#[derive(Debug, Clone)]
pub enum VariableValue {
    Scalar(String),
    IndexedArray(BTreeMap<usize, String>),
    AssociativeArray(BTreeMap<String, String>),
}

bitflags::bitflags! {
    #[derive(Debug, Clone, Copy, PartialEq, Eq)]
    pub struct VariableAttrs: u16 {
        const EXPORTED  = 0b0000_0001;
        const READONLY  = 0b0000_0010;
        const INTEGER   = 0b0000_0100;
        const LOWERCASE = 0b0000_1000;
        const UPPERCASE = 0b0001_0000;
        const NAMEREF   = 0b0010_0000;
    }
}

#[derive(Debug, Clone)]
pub struct Variable {
    pub value: VariableValue,
    pub attrs: VariableAttrs,
}
```

`VariableValue::Scalar` is the default. `BTreeMap<usize, String>` for indexed arrays preserves sparse semantics (bash arrays are sparse â€” `unset arr[5]` doesn't reindex). `BTreeMap<String, String>` for associative arrays gives deterministic iteration order for test stability.

### 2. Array Expansion Semantics

`${arr[@]}` vs `${arr[*]}` have subtly different behaviors: `"${arr[@]}"` produces one word per element (even with IFS), while `"${arr[*]}"` joins elements with the first character of `$IFS`. `${#arr[@]}` returns element count, not string length. `${!arr[@]}` returns keys, not values. These must integrate cleanly into the existing `expand_word` â†’ `resolve_parameter` pipeline in `expansion.rs`.

**Solution**: `resolve_parameter` gains array-aware branches for `Parameter::NamedWithIndex` and `Parameter::NamedWithAllIndices`. For `[@]` expansions, return a `Vec<String>` (multiple words) rather than a single string â€” this requires `expand_word` to handle multi-word returns from parameter resolution. Factor out a `ParameterResult` enum (`Single(String)` | `Multiple(Vec<String>)`) to carry this through cleanly.

### 3. Shopt Behavioral Wiring

Each shopt option must modify a different subsystem: `nullglob`/`failglob`/`dotglob`/`nocaseglob`/`globstar`/`globskipdots` affect glob expansion (in `expansion.rs` and `pattern.rs`), `nocasematch` affects `[[ =~ ]]` and `case` (in `walker.rs`), `lastpipe` changes pipeline execution (in `walker.rs`), `xpg_echo` changes the `echo` command (in `commands/mod.rs`). The shopt struct must be accessible from all these sites.

**Solution**: Add `ShoptOpts` to `InterpreterState` (it's already passed everywhere). Each behavioral site reads the relevant flag. The `shopt` builtin reads/writes the struct. No cross-cutting concerns â€” each flag is wired locally.

### 4. Process Substitution Without Real FDs

Real bash implements `<(cmd)` with `/dev/fd/N` backed by OS pipes. In a sandboxed VFS environment, there are no real FDs. We must simulate the behavior using temp VFS files.

**Solution**: For `<(cmd)`: execute command, capture stdout, write to temp VFS path `/tmp/.proc_sub_XXXXXX`, substitute that path into the argument list. For `>(cmd)`: create temp file, substitute its path, after outer command completes read the temp file and pipe contents as stdin to inner command. Clean up temp files with RAII guard. This works because commands already go through VFS for all file operations.

### 5. Nameref Depth Limiting

`declare -n ref=target` creates an indirection chain. Circular namerefs (`declare -n a=b; declare -n b=a`) must not infinite-loop. Real bash caps at about 10 levels.

**Solution**: `resolve_nameref(name, state, depth)` with `depth` capped at 10. Return an error on exceeding the cap: `"${name}: circular name reference"`.

## Phase 1 â€” Indexed and Associative Arrays (M6.1) âś…

> Extend Variable to hold array data. Handle array assignment, expansion, and arithmetic. Enforce maxArrayElements limit.
>
> **Status: COMPLETE.** All sub-tasks implemented and tested (31 integration tests). All 1609 tests pass.

This is the highest-impact phase â€” it unblocks M6.2, M6.4, M6.5, M6.6, M6.8, and M6.11.

### 1a. Evolve `Variable` struct

Add `VariableValue` enum, `VariableAttrs` bitflags, and update `Variable`:

```rust
// interpreter/mod.rs
use std::collections::BTreeMap;

#[derive(Debug, Clone, PartialEq)]
pub enum VariableValue {
    Scalar(String),
    IndexedArray(BTreeMap<usize, String>),
    AssociativeArray(BTreeMap<String, String>),
}

impl VariableValue {
    /// Return the scalar value, or element [0] for indexed arrays,
    /// or empty string for associative arrays (matches bash behavior).
    pub fn as_scalar(&self) -> &str { ... }

    /// Return element count for arrays, or 1 for non-empty scalars.
    pub fn count(&self) -> usize { ... }
}
```

Migrate `Variable.value: String` â†’ `Variable.value: VariableValue`, `Variable.exported: bool` / `Variable.readonly: bool` â†’ `Variable.attrs: VariableAttrs`. Update `set_variable()` helper and every site that reads `.value`, `.exported`, `.readonly`.

Add `maxArrayElements` to `ExecutionLimits` (default 100,000). Enforce on every array insert/append.

### 1b. Array assignment handling

In `walker.rs`, update the assignment handling code (currently at line ~290) to process:

- **`arr=(val1 val2 val3)`** â€” `AssignmentValue::Array` â†’ create `VariableValue::IndexedArray`
- **`arr[N]=val`** â€” `ArrayElementName(name, index)` â†’ insert into existing array at index, or create a new indexed array if the variable doesn't exist
- **`arr+=(val4 val5)`** â€” append to existing array (assign starting from `max_key + 1`)
- **`assoc[key]=val`** â€” if variable is declared as `AssociativeArray`, use string key

Arithmetic-evaluate index expressions (e.g., `arr[$((i+1))]=val`) using the existing `evaluate_arithmetic` function.

### 1c. Array parameter expansion

In `expansion.rs`, update `resolve_parameter` to handle:

- **`${arr[N]}`** (`Parameter::NamedWithIndex`) â€” look up element N in the array
- **`${arr[@]}`** / **`${arr[*]}`** (`Parameter::NamedWithAllIndices`) â€” expand to all values. `[@]` produces separate words; `[*]` joins with first char of IFS
- **`${#arr[@]}`** â€” element count (not string length)
- **`${!arr[@]}`** / **`${!arr[*]}`** (`ParameterExpr::MemberKeys`) â€” expand to all keys/indices
- **`${arr[@]:offset:length}`** â€” array slicing
- **`unset arr[N]`** â€” remove element without reindexing (sparse array semantics)
- **`unset arr`** â€” remove entire array variable

Introduce `ParameterResult` enum (`Single(String)` | `Multiple(Vec<String>)`) to carry multi-word expansions through `expand_word`. When `[@]` is inside double quotes, each element becomes a separate word (like `"$@"` for positional parameters).

### 1d. Array in arithmetic

In `arithmetic.rs`, support `arr[expr]` inside arithmetic expressions â€” evaluate the index expression, then look up the array element. If the variable is not an array, treat `var[0]` as the scalar value (bash compatibility).

### 1e. Tests

- Basic indexed array: `a=(one two three); echo ${a[0]} ${a[2]}` â†’ `one three`
- Sparse arrays: `a[0]=x; a[5]=y; echo ${#a[@]}` â†’ `2`
- All-elements: `a=(a b c); for x in "${a[@]}"; do echo "$x"; done` â†’ `a\nb\nc`
- Star expansion with IFS: `IFS=,; a=(a b c); echo "${a[*]}"` â†’ `a,b,c`
- Keys: `a=(x y z); echo ${!a[@]}` â†’ `0 1 2`
- Append: `a=(1 2); a+=(3 4); echo ${a[@]}` â†’ `1 2 3 4`
- Unset element: `a=(a b c); unset a[1]; echo ${a[@]}` â†’ `a c`
- Unquoted `${arr[@]}` with IFS characters in elements: elements containing IFS chars get split further
- maxArrayElements enforcement: loop creating > limit elements â†’ `LimitExceeded` error
- Arithmetic index: `a=(10 20 30); i=1; echo ${a[$((i+1))]}` â†’ `30`

## Phase 2 â€” Full `declare` Attributes (M6.6) âś…

> Extend declare and Variable to support all attribute flags: -i, -l, -u, -n, -a, -A.

### 2a. Attribute bitflags

The `VariableAttrs` bitflags type was added in Phase 1. Now wire the remaining flags into behavior:

- **`-i` (INTEGER)**: On every assignment to this variable, evaluate the value as an arithmetic expression. `declare -i x; x=2+3; echo $x` â†’ `5`.
- **`-l` (LOWERCASE)**: Transform value to lowercase on every assignment.
- **`-u` (UPPERCASE)**: Transform value to uppercase on every assignment.
- **`-n` (NAMEREF)**: Variable holds name of another variable; dereference on read/write.

### 2b. Integer attribute (-i)

In `set_variable()`, check if `INTEGER` flag is set. If so, pass the value through `evaluate_arithmetic()` and store the numeric result as a string. This applies to all assignment paths: plain assignment, `declare -i x=expr`, `read`, `for` loop variable.

### 2c. Case transform attributes (-l / -u)

In `set_variable()`, apply `.to_lowercase()` or `.to_uppercase()` before storing if the respective flag is set. The transform is on assignment, not on read.

### 2d. Nameref attribute (-n)

Add `resolve_nameref(name: &str, state: &InterpreterState, depth: usize) -> Result<&str, RustBashError>` that follows the nameref chain (up to depth 10). Wire into all variable read and write paths. `declare -n ref=target; ref=hello` sets `target=hello`. `echo $ref` prints the value of `target`.

### 2e. Array declaration (-a / -A)

`declare -a arr` creates an empty indexed array. `declare -A assoc` creates an empty associative array. When assigning to a variable declared as `-A`, use string keys instead of numeric indices.

### 2f. Attribute display

`declare -p varname` prints the declaration with all flags (e.g., `declare -ix var="5"`). `declare` with no args lists all variables with their attributes.

### 2g. Tests

- Integer: `declare -i x; x=2+3; echo $x` â†’ `5`
- Integer assignment propagation: `declare -i x=10; x+=5; echo $x` â†’ `15`
- Lowercase: `declare -l s; s=HELLO; echo $s` â†’ `hello`
- Uppercase: `declare -u s; s=hello; echo $s` â†’ `HELLO`
- Nameref: `x=42; declare -n ref=x; echo $ref` â†’ `42`; `ref=99; echo $x` â†’ `99`
- Nameref circular: `declare -n a=b; declare -n b=a; echo $a` â†’ error
- Declare -p: `declare -ix num=5; declare -p num` â†’ `declare -ix num="5"`
- Array declare: `declare -a arr; arr[0]=x; echo ${arr[0]}` â†’ `x`
- Associative declare: `declare -A m; m[hello]=world; echo ${m[hello]}` â†’ `world`
- Attribute conflict: `declare -lu var; var=Hello; echo $var` â†’ `HELLO` (last flag wins)

> Expose pipeline exit codes and regex captures as proper indexed arrays.

### 3a. PIPESTATUS

In `execute_pipeline()` (`walker.rs` line ~176), the `exit_codes: Vec<i32>` is already collected. After pipeline execution, write it as `PIPESTATUS` indexed array:

```rust
let mut map = BTreeMap::new();
for (i, code) in exit_codes.iter().enumerate() {
    map.insert(i, code.to_string());
}
set_array_variable(state, "PIPESTATUS", VariableValue::IndexedArray(map))?;
```

This enables `echo ${PIPESTATUS[0]}` and `echo ${PIPESTATUS[@]}`. For simple commands (not pipelines), set `PIPESTATUS=(exit_code)` with a single element. `PIPESTATUS` is overwritten by every pipeline or simple command â€” it always reflects the most recent command's status.


Remove the old `BASH_REMATCH_N` flat variables and `BASH_REMATCH_COUNT`.

## Phase 4 â€” Full `read` Flags (M6.5)

> Extend builtin_read beyond basic line reading.

### 4a. `-a arrayname` â€” read into indexed array

Split the input line by IFS and store each field as an array element. `echo "a b c" | read -a arr; echo ${arr[1]}` â†’ `b`.

### 4b. `-d delim` â€” read until delimiter

Instead of reading until newline, read until the specified delimiter character. `echo -n "hello:world" | read -d : var; echo $var` â†’ `hello`. Note: `read -d ''` (empty delimiter) means "read until NUL byte" â€” in practice this reads until EOF in our sandbox since VFS content rarely contains NUL. This is a commonly used pattern for reading entire files.

### 4c. `-n count` / `-N count` â€” character-limited read

`-n count`: read at most N characters (stop at newline or N, whichever first). `-N count`: read exactly N characters (ignore newlines). These differ in their newline handling.

### 4d. `-s` â€” silent mode (no-op in sandbox)

In interactive bash, `-s` suppresses echoing input to the terminal. In our sandbox there is no terminal, so accept the flag but take no special action. Scripts commonly use `read -s -p "Password: " pass`.

### 4e. `-t timeout` â€” timeout stub

In a sandbox, stdin is always fully provided. `-t` returns 0 immediately if there is data, 1 if stdin is exhausted. No actual timer needed.

### 4f. Tests

- Read into array: `echo "a b c" | read -ra arr; echo ${arr[1]}` â†’ `b`
- Delimiter: `printf 'a:b:c' | read -d : x; echo $x` â†’ `a`
- Character limit: `echo "hello" | read -n 3 x; echo $x` â†’ `hel`
- Exact count: `printf 'ab\ncd' | read -N 4 x; echo "$x"` â†’ `ab\nc` (newline embedded)
- Combined: `echo "x y z" | read -ra arr; echo ${#arr[@]}` â†’ `3`

## Phase 5 â€” Shopt Options (M6.3)

> Add ShoptOpts struct to InterpreterState and shopt builtin. Wire behavioral effects.

### 5a. ShoptOpts struct and builtin

Add `ShoptOpts` with boolean fields for each option to `InterpreterState`. Implement `shopt` builtin:
- `shopt -s opt` â€” enable option
- `shopt -u opt` â€” disable option
- `shopt -q opt` â€” query (exit 0 if set, 1 if unset)
- `shopt` with no args â€” list all options with on/off status
- `shopt -p` â€” print in reusable format (`shopt -s opt` or `shopt -u opt`)

### 5b. nullglob

When enabled, glob patterns that match nothing expand to nothing (empty list) instead of the literal pattern string. Modify the glob expansion path in `expansion.rs`.

### 5c. globstar

When enabled, `**` in glob patterns matches recursively across directory boundaries. Modify `pattern.rs` glob matching to support recursive descent.

### 5d. dotglob / globskipdots

`dotglob`: globs include dot-files (files starting with `.`). `globskipdots`: don't include `.` and `..` entries (bash 5.2+ default behavior). These modify the directory listing filter in glob expansion.

### 5e. failglob

When enabled, a glob that matches nothing produces an error instead of passing through as a literal. This is distinct from `nullglob` â€” `nullglob` removes the pattern, `failglob` errors.

### 5f. nocaseglob / nocasematch

`nocaseglob`: case-insensitive filename globbing. Modify pattern matching in `pattern.rs`. `nocasematch`: case-insensitive matching in `[[ str =~ pattern ]]` and `case` statements. Modify matching in `walker.rs`.

### 5g. lastpipe

When enabled (and job control is inactive, which is always true in our sandbox), the last command in a pipeline runs in the current shell context instead of a subshell. This means variable assignments in the last pipeline command persist. Modify `execute_pipeline` to skip the subshell wrapper for the final command when `lastpipe` is set.

**Note**: The current pipeline implementation in `walker.rs` runs all stages sequentially without subshell isolation, so `lastpipe` behavior may already be implicit. Verify with `echo hello | read x; echo $x` â€” if this already works, `lastpipe` is a tracked-but-no-op flag. If not, the final stage needs special handling to run in the parent shell context.

### 5h. expand_aliases / xpg_echo / extglob

`expand_aliases`: enable alias expansion (depends on M6.4 alias support). `xpg_echo`: make `echo` interpret backslash escapes by default (like `echo -e`). Wire into `echo` command in `commands/mod.rs`. `extglob`: extended glob patterns `+(...)`, `@(...)`, etc. â€” the parser already enables this unconditionally, so just track the flag and document that it defaults to on.

### 5i. Tests

- nullglob: `shopt -s nullglob; echo /nonexistent*` â†’ (empty line)
- failglob: `shopt -s failglob; echo /nonexistent*` â†’ error
- dotglob: `shopt -s dotglob; ls` includes dot-files
- nocasematch: `shopt -s nocasematch; [[ "Hello" == hello ]] && echo match` â†’ `match`
- globstar: `shopt -s globstar; echo **/*.txt` matches recursively
- lastpipe: `shopt -s lastpipe; echo hello | read x; echo $x` â†’ `hello`
- xpg_echo: `shopt -s xpg_echo; echo "a\tb"` â†’ `a	b`
- shopt -p output format

## Phase 6 â€” Additional Builtins (M6.4)

> Implement missing builtins that AI-generated scripts commonly use.

### 6a. `type [-t|-a|-p] name`

Identify whether `name` is a builtin, function, file, or alias. `type -t name` prints just the type word. `type -a name` shows all definitions. `type -p name` shows the file path (like `which`). Check `state.functions`, `state.commands`, and PATH resolution in order.

### 6b. `command [-pVv] name [args]`

`command name args` runs `name` bypassing functions (only builtins and commands). `command -v name` prints how `name` would be resolved (like `type -t`). `command -V name` provides verbose description. `command -p name` uses a default PATH. This is the most common tool-detection pattern: `if command -v git &>/dev/null; then ...`.

### 6c. `builtin name [args]`

Force execution of a shell builtin, bypassing any function with the same name. Look up `name` in the builtins table only â€” skip function lookup.

### 6d. `getopts optstring name [args]`

Argument parsing with `OPTIND`/`OPTARG`/`OPTERR` state variables. Process one option per call in a `while getopts` loop. Increment `OPTIND` to track position. Set `OPTARG` for options that take arguments (indicated by `:` after the option letter in optstring).

### 6e. `mapfile` / `readarray`

Populate an indexed array from stdin. Flags: `-t` (strip trailing newline), `-d delim` (delimiter, default newline), `-n count` (max lines), `-s count` (skip first N lines), `-C callback` (run callback every N lines). `readarray` is a synonym.

### 6f. `pushd` / `popd` / `dirs`

Full directory stack. Add `dir_stack: Vec<String>` to `InterpreterState`.
- `pushd dir` pushes current dir onto stack and cd's to dir. `pushd +N` rotates.
- `popd` pops top of stack and cd's there. `popd +N` removes Nth entry.
- `dirs` lists the stack. `-c` clears. `-l` shows full paths. `-p` one-per-line. `-v` with indices.

### 6g. `hash [-r] [name]`

Command path caching with `HashMap<String, String>` in `InterpreterState`. `hash name` resolves PATH and caches. `hash -r` clears the table. `hash` with no args lists entries. Cached paths are checked first during command resolution.

### 6h. `wait` â€” stub

No-op that returns 0. Prevents scripts from failing when they include `wait` for background job synchronization.

### 6i. `alias name=value` / `unalias name`

Add `aliases: HashMap<String, String>` to `InterpreterState`. `alias name=value` stores the mapping. `unalias name` removes it. `alias` with no args lists all. Alias expansion happens during word expansion: before command lookup, check if the first word is an alias and substitute. Guard against recursive expansion. Lower priority due to architectural complexity â€” real alias expansion should happen at the token level before parsing, but a post-parse first-word substitution is sufficient for common use cases. **Known limitation**: multi-word aliases (e.g., `alias ll='ls -la'`) require re-parsing the expanded tokens; the post-parse approach will handle simple single-command aliases correctly but may need enhancement for multi-word expansions.

### 6j. `select var in list; do ... done`

**Note**: `select` is a compound command (like `for`/`while`), not a simple builtin. brush-parser may produce a compound command AST node for it. Implementation goes in `execute_compound_command` in `walker.rs`, not in `builtins.rs`. In a sandbox (no interactive TTY), `select` reads from stdin. Print numbered options to stderr, read a line from stdin, set `var` to the selected option, set `REPLY` to the raw input. Loop until `break` or EOF. Low priority â€” verify brush-parser support before implementing.

## Phase 8 â€” Shell Option Enforcement (M6.9)

> Wire behavioral effects for set/shopt options that are currently parsed but ignored.

The current `ShellOpts` struct has only 4 fields (`errexit`, `nounset`, `pipefail`, `xtrace`). This phase adds `verbose`, `noexec`, `noclobber`, `allexport`, `noglob`, `posix`, `vi_mode`, and `emacs_mode` to the struct, and wires behavioral effects for all of them. The `set` builtin's option parsing must also be extended to handle these new flags.

### 8a. `set -x` (xtrace)

Before executing each simple command, if `xtrace` is true, write the expanded command to stderr prefixed with `$PS4` (default `"+ "`). Show expanded words, not raw source. Output goes to the stderr stream in the execution context.

### 8b. `set -v` (verbose)

Print each input line to stderr before parsing/expansion. This happens at the source-reading level â€” before the parser processes the line.

### 8c. `set -n` (noexec)

Parse but do not execute commands. Useful for syntax checking. The walker should skip execution of all command nodes when `noexec` is active. Note: `set` itself must always execute (otherwise `set +n` couldn't re-enable execution), so the noexec check happens in the walker, not in builtin dispatch.

### 8d. `set -o noclobber`

Prevent `>` from overwriting existing files. When `noclobber` is set and the target file exists, `>` returns an error: `"cannot overwrite existing file"`. The `>|` operator forces overwrite regardless.

### 8e. `set -a` (allexport)

Auto-export every variable on assignment. Check at every assignment site: `set_variable()`, `declare`, `local`, `for` loop variable, `read`. When `allexport` is active, set the `EXPORTED` flag on the variable.

### 8f. `set -f` (noglob)

Disable filename expansion (globbing). When active, glob patterns are treated as literal strings â€” skip the glob expansion step in `expand_word`.

### 8g. `set -o posix` â€” stub

Accept the option, store it. Initial behavioral change: special builtin errors become fatal (non-zero exit from `break`, `continue`, `return`, `export`, `unset`, `eval`, `set`, `shift`, `source` causes the shell to exit). Full POSIX semantics are future scope.

### 8h. `set -o vi` / `set -o emacs` â€” no-ops

Accept and track these options without any behavioral effect. They control line editing in interactive shells, which is not meaningful in a sandbox.

### 8i. Tests

- xtrace: `set -x; echo hello 2>&1` includes `+ echo hello`
- xtrace with pipeline: `set -x; echo a | cat 2>&1` shows trace for both commands
- noclobber: `set -C; echo a > file; echo b > file` â†’ error on second write
- noclobber override: `set -C; echo a > file; echo b >| file; cat file` â†’ `b`
- allexport: `set -a; X=1; env | grep X` â†’ `X=1`
- noglob: `set -f; echo *` â†’ literal `*`
- noexec: `set -n; echo hello` â†’ no output
- posix stub accepted without error

## Phase 9 â€” Advanced Redirections (M6.10)

> Implement missing redirection features.

### 9a. `exec` builtin

When invoked with only redirections (`exec > file`, `exec 3< file`), permanently redirect file descriptors for the rest of the shell session. This requires tracking persistent FD redirections in `InterpreterState`:

```rust
// New field on InterpreterState
pub(crate) persistent_fds: HashMap<i32, PersistentFd>,

pub(crate) enum PersistentFd {
    OutputFile(String),   // FD writes to this VFS path
    InputFile(String),    // FD reads from this VFS path
    DevNull,
    Closed,
}
```

The redirection application code (`apply_output_redirects`, `get_stdin_from_redirects`) must consult `persistent_fds` as a fallback when no per-command redirect is specified for an FD.

When invoked with a command (`exec cmd args`), replace the shell with the command â€” in our sandbox, just execute the command and exit with its status.

### 9b. `/dev/stdin`, `/dev/stdout`, `/dev/stderr`, `/dev/zero`, `/dev/full`

Special-case these paths in the redirection handling (`walker.rs`, currently only `/dev/null` is handled at line ~1240):
- `/dev/stdin` â†’ reads from the current stdin stream
- `/dev/stdout` â†’ writes to the current stdout stream
- `/dev/stderr` â†’ writes to the current stderr stream
- `/dev/zero` â†’ reads return empty bytes; writes are discarded
- `/dev/full` â†’ reads return empty; writes return ENOSPC error

### 9c. FD variable allocation `{varname}>file`

Automatically allocate a file descriptor number (starting from 10, incrementing), store the number in the named variable. Used for advanced I/O multiplexing patterns. Requires extending the redirection parser handling to recognize the `{varname}` prefix.

### 9d. Read-write FD `N<>file`

Open a file for both reading and writing on FD N. Add a new redirection variant handling in the walker.

### 9e. FD movement `N>&M-`

Duplicate FD M to N, then close M. The `-` suffix signals the close-after-dup semantics.

### 9f. Pipe stderr `|&`

Shorthand for `2>&1 |`. When the parser produces a `|&` pipe, treat it as piping both stdout and stderr to the next command.

### 9g. Tests

- `exec > file; echo hello; cat file` â†’ `hello`
- `cat /dev/null` â†’ empty; `echo test > /dev/null` â†’ no output
- `echo hello > /dev/stderr 2>&1` â†’ `hello` on stderr

## Phase 10 â€” Parameter Transformation Operators (M6.11)

> Implement ${var@operator} syntax, indirect expansion, and printf -v.

### 10a. `${var@Q}`, `${var@E}`, `${var@P}`, `${var@A}`, `${var@a}`

Update `apply_transform` in `expansion.rs` (currently at line ~1016):

- **`@Q`**: Quote value for shell reuse. Simple values get single-quoted; values with control characters use `$'...'` notation.
- **`@E`**: Expand backslash escape sequences (`\n`, `\t`, `\xHH`, `\uHHHH`, etc.).
- **`@P`**: Expand prompt escape sequences (`\u` â†’ username, `\h` â†’ hostname, `\w` â†’ cwd, `\d` â†’ date, `\t` â†’ time, `\[`/`\]` â†’ empty). Used for PS1/PS4 expansion.
- **`@A`**: Produce assignment statement recreating the variable (e.g., `declare -- var="value"` or `declare -a arr='(1 2 3)'`). Include declare flags based on `Variable.attrs`.
- **`@a`**: Return attribute flags as a string (e.g., `"x"` for exported, `"r"` for readonly, `"xi"` for exported+integer). Read from `Variable.attrs`.

### 10b. `${!ref}` â€” indirect expansion

The expansion pipeline already has `indirect: bool` plumbing in `resolve_parameter` (expansion.rs) that dereferences one level. The remaining work is:
- Handle `${!arrref[@]}` where the indirection target is an array (expand to all elements of the referenced array)
- Handle `${!arrref[N]}` where indirection resolves an array element
- Ensure interaction with namerefs (M6.6) is correct: `${!ref}` is read-time expansion, namerefs are variable-level redirection â€” they are independent mechanisms

`x=hello; y=x; echo ${!y}` â†’ `hello`. Different from namerefs â€” this is a read-time expansion only.

### 10c. `${!prefix*}` / `${!prefix@}` â€” variable name expansion

Expand to all variable names matching the given prefix. `DOCKER_A=1; DOCKER_B=2; echo ${!DOCKER_*}` â†’ `DOCKER_A DOCKER_B`. Iterate `state.env` keys with the prefix filter.

### 10d. `printf -v varname` and format specifiers `%b`, `%q`

`printf` is currently an external command (`PrintfCommand` in `commands/text.rs`). To support `-v` (which needs to write to `InterpreterState`), add `printf` as a shell builtin that intercepts `-v varname`. The builtin parses the `-v` flag: if present, it calls the existing `PrintfCommand` formatting logic (extracted into a shared function) to produce the output string, then assigns it to the named variable via `set_variable()` instead of printing to stdout. Non-`-v` calls delegate directly to the existing formatting logic with normal stdout output. This avoids duplicating the ~200 lines of format handling.

- **`%b`**: Like `%s` but interpret backslash escape sequences in the argument.
- **`%q`**: Shell-quote the argument so it can be safely reused as input.

## Phase 11 â€” Process Substitution (M6.7)

> Implement <(cmd) and >(cmd) using temp VFS files.

### 11a. `<(cmd)` â€” input process substitution

When the parser produces a `ProcessSubstitution::Read` node:
1. Execute the inner command, capturing stdout
2. Write captured output to a temp VFS file (`/tmp/.proc_sub_{counter}`)
3. Substitute the temp file path into the argument list

The outer command sees a regular file path and reads from it normally. This enables patterns like `diff <(sort file1) <(sort file2)`.

### 11b. `>(cmd)` â€” output process substitution

When the parser produces a `ProcessSubstitution::Write` node:
1. Create an empty temp VFS file
2. Substitute its path into the argument list
3. After the outer command completes, read the temp file contents
4. Execute the inner command with the file contents as stdin

**Known limitation**: This is a batch model (write all, then read all), not streaming. In real bash, `>(cmd)` uses a pipe so the inner command sees data incrementally. In our sandbox everything is in-memory so the functional result is the same, but timing-sensitive scripts (rare) may behave differently.

### 11c. Temp file cleanup

Use an RAII guard or explicit cleanup after the enclosing command completes. Track active temp files in a list, remove them after the statement finishes. Add a counter to `InterpreterState` for unique temp file naming.

- `echo hello > >(cat)` â†’ `hello`
- `paste <(echo a) <(echo b)` â†’ `a	b`
- Temp files cleaned up after command completes
- Nested: `cat <(cat <(echo deep))` â†’ `deep`
