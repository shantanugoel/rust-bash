# Milestone 1: Core Shell ‚Äî Implementation Plan

## Problem Statement

Build a correct, reliable sandboxed bash interpreter from scratch in Rust. The project is currently a blank slate (`lib.rs` has only a placeholder). M1 delivers: a `VirtualFs` trait + `InMemoryFs`, a full interpreter engine (AST walking, word expansion, pipelines, control flow), ~35 commands, and a public API (`RustBash::exec()`).

## Approach

Follow the guidebook's recommended order (Ch. 10) but reorganize into pragmatic phases since we're bootstrapping from zero ‚Äî not "extracting" or "fixing" existing code. Each phase is testable end-to-end before moving to the next.

Dependencies: `brush-parser` (pinned git rev), `parking_lot` (non-poisoning RwLock), `insta` (snapshot testing, dev-dep).

## Phase 1 ‚Äî Interpreter Skeleton & Public API (M1.2 + M1.15-partial)

> Minimal end-to-end: `shell.exec("echo hello")` returns `ExecResult { stdout: "hello\n", .. }`.
> Split into sub-phases for manageable increments.

### Phase 1A ‚Äî Core Types, Parsing & Minimal Execution

#### 1A-a. Core types
- `Variable` struct: `{ value: String, exported: bool, readonly: bool }`
- `CommandResult`: `{ stdout: String, stderr: String, exit_code: i32 }`
- `VirtualCommand` trait: `name() -> &str`, `execute(&[String], &CommandContext) -> CommandResult`
- `CommandContext`: `{ fs, cwd, env, stdin, limits, exec }`
  - `limits: &ExecutionLimits` ‚Äî stub with defaults, no enforcement yet (enforcement in M3)
  - `exec` callback is `None` initially (M1.5 adds it)
- `ExecutionLimits` struct with all fields from Ch. 7 and sensible defaults (via `Default` impl). No enforcement logic yet ‚Äî just carry the struct so type signatures are correct from the start.
- `ExecutionCounters` struct: `{ command_count, call_depth, output_size, start_time }`. Reset per `exec()` call.
- `InterpreterState`: `{ fs, env, cwd, functions, last_exit_code, commands, shell_opts, limits, counters }`
  - `shell_opts` tracks `set -e`, `set -u`, `set -o pipefail` (flags only, enforcement in M1.15)
- `ExecResult`: `{ stdout: String, stderr: String, exit_code: i32 }`
- `RustBash` struct owning `InterpreterState`
- `RustBashBuilder` with `.files()`, `.env()`, `.cwd()`, `.command()`, `.execution_limits()`, `.build()`

#### 1A-b. Parsing pipeline
- Integrate `brush_parser::tokenize_str()` + `brush_parser::parse_tokens()`
- Parser options: bash mode, extended globbing enabled
- Wrap parse errors in `RustBashError::Parse`

#### 1A-c. AST walker ‚Äî compound lists, and-or lists, pipelines
- `execute_program(program, state) -> Result<ExecResult>`
- `execute_compound_list(list, state)` ‚Äî iterate items, **accumulate** stdout/stderr (M1.2)
- `execute_and_or_list(aol, state)` ‚Äî short-circuit `&&`/`||` based on exit code
- `execute_pipeline(pipeline, state)` ‚Äî chain stdout‚Üístdin between commands; handle `!` negation
- Separator handling: `;` (sequential), `&` (treat as sequential ‚Äî no background processes per Ch. 1 non-goals)

#### 1A-d. Simple command execution
- `execute_simple_command(cmd, state)` ‚Äî expand words, resolve command, dispatch
- Command resolution order (per Ch. 4):
  1. **Special shell builtins** ‚Äî state-modifying, unshadowable: `cd`, `export`, `unset`, `set`, `exit`, `local`, `return`, `break`, `continue`, `eval`, `source`, `read`, `shift`, `declare`, `readonly`, `trap`
  2. **User-defined functions** ‚Äî from `state.functions`
  3. **Registered commands** ‚Äî from `state.commands` HashMap (includes `echo`, `test`, `cat`, etc.). **Can be shadowed by functions.**
  4. **"Command not found"** ‚Äî stderr error, exit code 127
- Pre-command variable assignments (e.g., `FOO=bar cmd`):
  - With a command: set `FOO` only for duration of `cmd` (not persisted)
  - Without a command (`FOO=bar` alone): persisted in environment
  - Note: `FOO=bar echo $FOO` does NOT expand to "bar" because expansion happens before the assignment takes effect

#### 1A-e. Minimal builtins & commands
- `true` / `false` ‚Äî exit 0 / exit 1 (builtins for performance)
- `echo` ‚Äî `-n` (no newline), `-e` (escape sequences). Registered as command (shadowable by functions).
- `exit` ‚Äî return exit code

**Exit criteria for 1A**: `shell.exec("echo hello")` ‚Üí `ExecResult { stdout: "hello\n", exit_code: 0 }`. `cargo test` + `cargo clippy` clean.

### Phase 1B ‚Äî Word Expansion, Redirections & Builtins

#### 1B-a. Basic word expansion
- Integrate `brush_parser::word::parse()` for word decomposition
- Handle `WordPiece` types: `Text`, `SingleQuotedText`, `DoubleQuotedSequence`, `ParameterExpansion`, `TildePrefix`
- Basic parameter expansion: `$VAR`, `${VAR}`, `${VAR:-default}`, `${VAR:=default}`, `${VAR:?msg}`, `${VAR:+alt}`
- Advanced parameter expansion (all from Ch. 4):
  - `${#VAR}` ‚Äî string length
  - `${VAR%pattern}` / `${VAR%%pattern}` ‚Äî suffix removal (shortest/longest)
  - `${VAR#pattern}` / `${VAR##pattern}` ‚Äî prefix removal (shortest/longest)
  - `${VAR/pattern/string}` ‚Äî substitution (first match); `${VAR//pattern/string}` (all matches)
  - `${VAR:offset:length}` ‚Äî substring extraction
  - `${VAR^}` / `${VAR^^}` / `${VAR,}` / `${VAR,,}` ‚Äî case modification
- Special variables: `$?`, `$#`, `$@`, `$*`, `$0`, `$1`‚Äì`$9`, `${10}+`, `$$`, `$!` (always empty), `$RANDOM`, `$LINENO`
- Tilde expansion: `~` ‚Üí `$HOME`
- **No** IFS word splitting yet (Phase 2), **no** command substitution yet (Phase 3), **no** globs yet (Phase 7a)

#### 1B-b. Redirections
- `> file` (write stdout), `>> file` (append stdout), `< file` (read stdin)
- `2> file`, `2>> file` (stderr redirect)
- `2>&1` (stderr‚Üístdout), `&> file` (stdout+stderr to file)
- `/dev/null` support (discard output)

#### 1B-c. Shell builtins (full first batch)
- `cd` ‚Äî change `state.cwd`, validate path exists and is dir in VFS
- `export` ‚Äî mark variable as exported; `export VAR=value` sets and exports
- `unset` ‚Äî remove variable
- `set` ‚Äî parse flags (`-e`, `-u`, `-o pipefail`, `-x`), `set --` for positional params
- `shift` ‚Äî shift positional parameters
- `declare` / `readonly` ‚Äî variable attributes
- `read` ‚Äî read line from stdin into variable(s). Shell builtin because it modifies `state.env`.
  - `-r` (no backslash escaping), `-p` (prompt, no-op in sandbox), IFS-based field splitting

**Exit criteria for 1B**: Variable expansion, redirections, and builtins work. `echo $HOME > /file && cat /file` works.

### Phase 1C ‚Äî Compound Commands & Starter Commands

#### 1C-a. Compound commands
- `if`/`elif`/`else` ‚Äî evaluate condition lists, execute matching body
- `for var in words; do body; done` ‚Äî expand word list, iterate
- `while condition; do body; done` / `until condition; do body; done`
- Brace groups `{ cmds; }` ‚Äî execute in current scope
- Subshell `( cmds )` ‚Äî **deep-clone** state, execute, discard state changes
  - VFS: `InMemoryFs` needs a true deep clone (not just `Arc` clone, which shares the tree)
  - env, functions, cwd: all cloned, changes discarded after subshell exits
  - Exit code: propagates back (it's the subshell's return value)
  - Test: `(cd /tmp && echo $PWD); echo $PWD` ‚Äî cwd should NOT change outside subshell

#### 1C-b. Remaining starter commands (registered commands)
- `cat` ‚Äî concatenate files/stdin, `-n` (line numbers)
- `touch` ‚Äî create empty file or update mtime
- `mkdir` ‚Äî `-p` (parents)
- `ls` ‚Äî `-l`, `-a`, `-1`, `-R`
- `pwd` ‚Äî print working directory

### Phase 1D ‚Äî Integration Tests

- `echo hello` ‚Üí stdout "hello\n"
- `echo a; echo b` ‚Üí stdout "a\nb\n" (compound list accumulation ‚Äî M1.2 verified)
- `echo hello | cat` ‚Üí stdout "hello\n"
- Redirections: `echo hello > /file.txt && cat /file.txt`
- State persistence: two `exec()` calls, verify VFS/env/cwd carry over
- `if true; then echo yes; else echo no; fi` ‚Üí "yes\n"
- `for i in a b c; do echo $i; done` ‚Üí "a\nb\nc\n"
- `FOO=bar; echo $FOO` ‚Üí "bar\n"
- Pre-command assignment: `FOO=bar echo done; echo $FOO` ‚Üí `done\n\n` (FOO not persisted)
- Subshell isolation: `X=outer; (X=inner; echo $X); echo $X` ‚Üí `inner\nouter\n`
- Error cases: parse error, command not found (exit 127)
- Snapshot tests (via `insta`) for `ls` and other commands with structured output

**Exit criteria**: Basic shell works end-to-end. `cargo test` + `cargo clippy` clean.

## Phase 2 ‚Äî Word Splitting & Quoting (M1.3)

### 2a. IFS-based word splitting
- After parameter expansion (and later command substitution/arithmetic), split unquoted results on `$IFS`
- Default IFS: space, tab, newline
- Custom IFS support (e.g., `IFS=: read` patterns)
- Consecutive IFS whitespace collapses; non-whitespace IFS chars produce empty fields

### 2b. Quoting correctness
- Double-quoted `"$VAR"` ‚Äî expand but do NOT word-split
- Single-quoted `'$VAR'` ‚Äî literal, no expansion
- `"$@"` ‚Äî expands to separate words, one per positional param
- `"$*"` ‚Äî expands to single word joined by first char of IFS
- Unquoted `$@` and `$*` ‚Äî expand then word-split

### 2c. Tests
- `VAR="a b c"; for w in $VAR; do echo $w; done` ‚Üí `a\nb\nc\n`
- `VAR="a b c"; for w in "$VAR"; do echo $w; done` ‚Üí `a b c\n`
- `set -- x y z; for w in "$@"; do echo $w; done` ‚Üí `x\ny\nz\n`
- IFS override tests
- Edge cases: empty variables, unset variables, variables with newlines

## Phase 4 ‚Äî Exec Callback (M1.5)

> Does NOT depend on Phase 3 (command substitution). eval/source just need recursive interpreter invocation, which exists after Phase 1.

### 4a. Exec callback on CommandContext
- Add `exec: Option<&dyn Fn(&str) -> CommandResult>` to `CommandContext`
- Wire up in interpreter: commands that need sub-execution receive the callback
- The callback parses and executes a string through the interpreter

### 4b. eval builtin
- `eval "string"` ‚Äî concatenate args, parse, execute in current shell context
- Variable expansion happens before eval sees the string

### 4c. source builtin
- `source /path/to/file` (or `. /path/to/file`) ‚Äî read file from VFS, parse, execute
- Executes in current shell context (variables persist)

### 4d. Tests
- `eval 'echo hello'` ‚Üí `hello\n`
- `eval "echo \$HOME"` ‚Üí value of HOME
- Write script to VFS, `source` it, verify side effects
- `eval` with variable assignments that persist

## Phase 6 ‚Äî Loop Control (M1.7)

> Independent of Phase 5 (test/[[). These two phases can be built in parallel.

### 6a. break/continue signals
- Define control flow signal type: `ControlFlow::Break(depth)`, `ControlFlow::Continue(depth)`
- `break` / `break N` ‚Äî exit N enclosing loops
- `continue` / `continue N` ‚Äî skip to next iteration of Nth enclosing loop
- Signal propagates up through compound command handlers

### 6b. Integration into loop handlers
- For/while/until handlers catch `Break`/`Continue` signals
- Decrement depth counter as signal passes through each loop level
- `break 0` and `continue 0` are errors

### 6c. Tests
- `for i in 1 2 3; do if [ $i = 2 ]; then break; fi; echo $i; done` ‚Üí `1\n`
- `for i in 1 2 3; do if [ $i = 2 ]; then continue; fi; echo $i; done` ‚Üí `1\n3\n`
- Nested: `break 2` exits two levels
- `break` outside loop ‚Üí error

## Phase 7 ‚Äî Expansion Features (M1.8, M1.9, M1.10, M1.11)

> These four sub-milestones are independent of each other and can be built in any order.

### 7a. Glob expansion (M1.8)
- Implement `VirtualFs::glob()` on `InMemoryFs` ‚Äî tree walk matching pattern
- Glob patterns: `*`, `?`, `[abc]`, `[a-z]`, `[!abc]`, `**` (recursive)
- Integrate into word expansion: unquoted words with glob chars ‚Üí expand against VFS
- If no match, leave pattern as literal (bash default without `failglob`)
- Simple numeric guard: cap results at a configurable maximum (e.g., 100,000)
- Tests: `echo *.txt`, `echo /dir/**/*.md`, no-match passthrough

### 7b. Brace expansion (M1.9)
- `{a,b,c}` ‚Üí three words: `a`, `b`, `c`
- `{1..5}` ‚Üí `1 2 3 4 5`
- `{1..10..2}` ‚Üí `1 3 5 7 9` (step)
- `{a..z}` ‚Üí letter sequences
- Nested: `{a,b{1,2}}` ‚Üí `a b1 b2`
- Combined with other text: `file{1,2,3}.txt` ‚Üí `file1.txt file2.txt file3.txt`
- Numeric guard against unbounded expansion
- Brace expansion happens before all other expansions
- Tests: various patterns, combined with prefix/suffix, nesting

### 7c. Here-documents and here-strings (M1.10)
- `<<EOF ... EOF` ‚Äî multi-line stdin from heredoc body (already in AST)
- `<<-EOF` ‚Äî strip leading tabs from body
- Quoted delimiter (`<<'EOF'`) ‚Üí no variable expansion in body
- Unquoted delimiter ‚Üí perform variable expansion in body
- `<<<word` ‚Äî expand word, use as stdin (with trailing newline)
- Tests: heredoc with/without expansion, here-string, indented heredoc

### 7d. Arithmetic expansion (M1.11)
- `$((...))` evaluator with full operator support:
  - Arithmetic: `+`, `-`, `*`, `/`, `%`, `**` (power)
  - Comparisons: `<`, `>`, `<=`, `>=`, `==`, `!=`
  - Boolean: `&&`, `||`, `!`
  - Bitwise: `&`, `|`, `^`, `~`, `<<`, `>>`
  - Ternary: `cond ? true_val : false_val`
  - Assignment: `=`, `+=`, `-=`, `*=`, `/=`, `%=`
  - Pre/post increment/decrement: `++var`, `var++`, `--var`, `var--`
- Variable references in expressions: `$((x + y))` reads vars from env
- `let` builtin: `let "x = 5 + 3"`
- `(( expr ))` compound command ‚Äî evaluates expression, exit code 0 if non-zero result
- `for (( init; cond; step )) do body; done` ‚Äî C-style arithmetic for loop (AST: `ArithmeticForClause`)
- Tests: all operators, variable references, nested expressions, division by zero, C-style for loop

## Phase 8 ‚Äî Functions & Local Variables (M1.12)

> Soft dependency on Phase 5 (test) ‚Äî functions themselves only need the interpreter skeleton; richer test cases use test/[[. Hard dependency on Phase 6 (break/continue) for the `return` signal mechanism.

### 8a. Function definitions
- `FunctionDef` struct: name + body (CompoundCommand from AST)
- Store in `state.functions` HashMap
- Both syntaxes: `func() { ... }` and `function func { ... }`

### 8b. Function calls
- Lookup in `state.functions` (after builtins, before registered commands)
- Save/restore positional parameters (`$1`, `$2`, etc.)
- Call depth tracking (for future limits)

### 8c. local and return
- `local var=value` ‚Äî creates function-scoped variable; restores previous value on return
- `return N` ‚Äî exit function with code N; uses control flow signal mechanism
- `return` without args ‚Üí uses `$?`

### 8d. Variable scoping
- Dynamic scoping (bash behavior): called functions see caller's variables
- `local` masks the caller's variable within the function scope
- `export` marks for child process inheritance (maintained for compatibility)

### 8e. Tests
- Define and call function, verify output
- Positional parameters in function
- `local` variable scoping: doesn't leak to caller
- Recursive function (with depth tracking)
- `return` with exit code
- Function shadowing a command name

### Phase 10-late ‚Äî Commands needing advanced interpreter features

## Phase 11 ‚Äî Error Handling Polish (M1.15)

### 11a. RustBashError finalization
- Ensure all variants are populated: Parse, Execution, LimitExceeded, Vfs, Timeout
- Implement `std::error::Error` + `Display` for all error types
- Ensure all public APIs return `Result<T, RustBashError>`

### 11b. set -e (errexit)
- Exit immediately when a command fails (non-zero exit code)
- Exceptions: conditions in `if`/`while`/`until`, left side of `&&`/`||`, negated commands

### 11c. set -u (nounset)
- Error on expansion of unset variables
- Exception: `${VAR:-default}` and similar default-value expansions

### 11d. set -o pipefail
- Pipeline exit code = exit code of rightmost failed command (not just last command)

### 11e. Tests
- `set -e` with failing command stops execution
- `set -e` with `if cmd_that_fails` does NOT stop
- `set -u` with unset variable ‚Üí error
- `set -o pipefail` with `false | true` ‚Üí exit 1

**Exit criteria**: Full M1 test suite passes. `cargo fmt`, `cargo clippy -- -D warnings`, `cargo test` all clean.

## Notes

- All file operations go through `VirtualFs` ‚Äî never `std::fs`
- No `std::process::Command` ‚Äî all commands are in-process Rust
- brush-parser handles all parsing; we only interpret
- Start sync API; async wrapper is a later concern
- Error messages follow format: `command: message` (not exact bash wording)
- Dynamic variable scoping matches bash
- `&` (background separator) treated as sequential execution ‚Äî no background processes
- `read` is a shell builtin (not a registered command) because it modifies interpreter state
- All registered commands support `--` flag terminator per Ch. 6 conventions
- Fuzz target (`cargo fuzz`) should be added once Phase 3 is complete (parser‚Üíinterpreter boundary is the key attack surface)
