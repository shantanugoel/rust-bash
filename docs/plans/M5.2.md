# Milestone 5.2: C FFI — Implementation Plan

## Problem Statement

rust-bash is a fully-featured sandboxed bash interpreter (80 commands, 4 filesystem backends, execution limits, network policy) available only as a Rust library crate. To enable embedding from Python, Go, Ruby, C, C++, or any language with C interop capabilities, we need a stable C-compatible ABI. This is one of the most impactful integration targets — nearly every language has C FFI support.

M5.2 delivers:
- **Shared library** (`.so` / `.dylib` / `.dll`) with a stable C ABI
- **Six exported functions**: `rust_bash_create`, `rust_bash_exec`, `rust_bash_result_free`, `rust_bash_free`, `rust_bash_last_error`, `rust_bash_version`
- **JSON-based configuration** for language-agnostic sandbox setup
- **Generated C header** (`include/rust_bash.h`) via `cbindgen`
- **Cross-language examples**: Python (ctypes) and Go (cgo)

Chapter 8 of the guidebook specifies the target C API interface.

## M1–M4 Carryover

All previous milestones (M1–M4) are complete. The Rust API (`RustBashBuilder`, `RustBash::exec()`, `ExecResult`) is stable and provides the foundation that the FFI layer wraps. No open items need to be addressed.

## Key Design Decisions

### 1. Feature-gated FFI

The FFI module is gated behind an `ffi` Cargo feature. Library-only Rust consumers don't need FFI symbols in their build.

```toml
[features]
ffi = ["dep:serde"]

[lib]
crate-type = ["lib", "cdylib"]
```

The FFI module is conditionally compiled:
```rust
#[cfg(feature = "ffi")]
pub mod ffi;
```

Build the shared library: `cargo build --features ffi --release`

> **Note on crate-type:** `crate-type` includes `cdylib` unconditionally because Cargo does not support conditional crate types via features. When the `ffi` feature is off, the cdylib is produced but exports no `extern "C"` symbols. The additional link time can be 5–15 seconds in debug builds. If `cargo test` cycle time degrades noticeably, the FFI layer should be extracted into a separate `rust-bash-ffi/` workspace crate.

If M5.1 has already been implemented and added a `[features]` section, append `ffi` to it. If not, create the section fresh.

### 2. JSON Configuration

Configuration is passed as a JSON string for maximum language interop. This avoids exposing Rust's builder pattern (which doesn't map naturally to C). The JSON schema matches Chapter 8:

```json
{
  "files": {
    "/data.txt": "content",
    "/config.json": "{}"
  },
  "env": {
    "USER": "agent",
    "HOME": "/home/agent"
  },
  "cwd": "/",
  "limits": {
    "max_command_count": 10000,
    "max_execution_time_secs": 30
  },
  "network": {
    "enabled": true,
    "allowed_url_prefixes": ["https://api.example.com/"]
  }
}
```

All fields are optional. An empty JSON object `{}` or a `NULL` pointer both create a default sandbox.

### 3. Memory Ownership Model

Standard C create/free ownership pattern:
- `rust_bash_create` → heap-allocated sandbox; caller owns it, must call `rust_bash_free`
- `rust_bash_exec` → heap-allocated `CExecResult`; caller owns it, must call `rust_bash_result_free`
- String data in `CExecResult` is owned by the result struct — pointers are valid until `rust_bash_result_free` is called
- `rust_bash_last_error` returns a pointer to a thread-local string — valid until the next FFI call on the same thread

### 4. Error Handling via Thread-Local Last-Error

Functions that can fail return `NULL` on error. The caller retrieves the error message via `rust_bash_last_error()`, which returns a pointer to a null-terminated thread-local string (or `NULL` if no error occurred).

```c
RustBash* sb = rust_bash_create("{invalid json}");
if (sb == NULL) {
    fprintf(stderr, "Error: %s\n", rust_bash_last_error());
}
```

This pattern is used by many C libraries (OpenSSL, SQLite, etc.) and is straightforward for all FFI consumers.

### 5. `#[repr(C)]` Result Struct with Pointer+Length Strings

The `CExecResult` struct uses `#[repr(C)]` for ABI stability. Strings are represented as pointer+length pairs (not null-terminated) to faithfully represent output that may theoretically contain null bytes.

```c
typedef struct CExecResult {
    const char* stdout_ptr;
    int32_t stdout_len;
    const char* stderr_ptr;
    int32_t stderr_len;
    int32_t exit_code;
} CExecResult;
```

Internally, each string is heap-allocated as a `Box<[u8]>` (via `Box::into_raw`). The `rust_bash_result_free` function reclaims both the strings and the struct.

### 6. Thread Safety

`RustBash` requires `&mut self` for `exec()`. The FFI layer does NOT add synchronization — callers must ensure that a single `RustBash*` handle is not used concurrently from multiple threads. This is documented in the header file and all examples.

### 7. Custom Commands via Function Pointers (deferred to M5.4)

Chapter 8 / M5.4 mentions custom command dispatch via C function pointers. This is explicitly out of scope for M5.2 — it will be addressed in M5.4 when the broader custom command callback pattern is finalized across all integration targets (C FFI, WASM, napi-rs).

### 8. Panic Safety Across the FFI Boundary

Unwinding a Rust panic across an `extern "C"` boundary is **undefined behavior**. Every FFI entry point must be wrapped in `std::panic::catch_unwind` to convert panics into a `set_last_error` + `NULL` return.

```rust
#[unsafe(no_mangle)]
pub extern "C" fn rust_bash_exec(sb: *mut RustBash, command: *const c_char) -> *mut CExecResult {
    let result = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
        // actual implementation
    }));
    match result {
        Ok(v) => v,
        Err(_) => {
            set_last_error("Internal panic in rust_bash_exec".into());
            std::ptr::null_mut()
        }
    }
}
```

All six `extern "C"` functions in Phase 2 must use this pattern. The `AssertUnwindSafe` wrapper is needed because `&mut RustBash` is not `UnwindSafe` — this is acceptable since a panic leaves the sandbox in an indeterminate state, and the caller should `rust_bash_free` it.

### 9. Binary File Limitation

The JSON config represents file contents as JSON strings (`"files": {"/path": "text content"}`). This means binary files (images, protobuf, compiled artifacts) cannot be seeded via the JSON config. This is a known limitation of the JSON approach. A future enhancement could accept base64-encoded content with a prefix convention (e.g., `"base64:AAEC..."`), but this is out of scope for M5.2.

### 10. Rust 2024 Edition: `#[unsafe(no_mangle)]`

This project uses Rust edition 2024, which requires the `#[unsafe(no_mangle)]` attribute form instead of the older `#[no_mangle]`. All FFI functions must use `#[unsafe(no_mangle)]`.

## Phase 2 — Core FFI Functions ✅

> Implement all public `extern "C"` functions: create, exec, free, result_free, last_error, version. Every function must be wrapped in `std::panic::catch_unwind` (see Key Design Decision §8) and annotated with `#[unsafe(no_mangle)]` (Rust 2024 edition, see §10).

### 2a. Implement thread-local error storage

```rust
thread_local! {
    static LAST_ERROR: RefCell<Option<CString>> = RefCell::new(None);
}
```

Two internal helpers:
- `set_last_error(msg: String)` — stores a `CString` in the thread-local
- `clear_last_error()` — clears the thread-local (called at the start of each FFI function)

Public function:
- `rust_bash_last_error() -> *const c_char` — returns the stored error string pointer, or `NULL` if no error

### 2b. Define `CExecResult` struct

```rust
#[repr(C)]
pub struct CExecResult {
    pub stdout_ptr: *const c_char,
    pub stdout_len: i32,
    pub stderr_ptr: *const c_char,
    pub stderr_len: i32,
    pub exit_code: i32,
}
```

### 2c. Implement `rust_bash_create`

```c
// C signature:
RustBash* rust_bash_create(const char* config_json);
```

Behavior:
1. Clear last error
2. If `config_json` is `NULL`, use default config
3. Validate UTF-8, parse JSON into `FfiConfig`
4. Build `RustBash` via `build_from_config()`
5. On success: return `Box::into_raw(Box::new(shell))`
6. On error: set last error, return `NULL`

### 2d. Implement `rust_bash_exec`

```c
// C signature:
CExecResult* rust_bash_exec(RustBash* sb, const char* command);
```

Behavior:
1. Clear last error
2. Null-check `sb` and `command`; set error and return `NULL` if null
3. Validate `command` as UTF-8
4. Call `shell.exec(cmd_str)`
5. On success: allocate `CExecResult` with stdout/stderr as heap `Box<[u8]>` via `Box::into_raw`
6. On `RustBashError`: set last error, return `NULL`

String allocation detail: convert `String` → `Vec<u8>` → `Box<[u8]>` → `Box::into_raw()` to get the pointer. Store the `len` alongside. This avoids null-termination issues and correctly handles any byte content.

**Important — fat→thin pointer conversion:** `Box::into_raw(Box<[u8]>)` returns `*mut [u8]` (a fat pointer with embedded length), but `CExecResult` stores `*const c_char` (a thin pointer). The correct conversion sequence is:

```rust
let bytes: Vec<u8> = result.stdout.into_bytes();
let len = bytes.len() as i32;
let boxed: Box<[u8]> = bytes.into_boxed_slice();
let fat_ptr: *mut [u8] = Box::into_raw(boxed);
let thin_ptr: *const c_char = fat_ptr as *mut u8 as *const c_char;
```

And in `rust_bash_result_free`, reconstruct the fat pointer for deallocation:
```rust
let fat_ptr = std::ptr::slice_from_raw_parts_mut(ptr as *mut u8, len as usize);
drop(unsafe { Box::from_raw(fat_ptr) });
```

This is safe because `into_boxed_slice()` guarantees `len == capacity`.

### 2e. Implement `rust_bash_result_free`

```c
// C signature:
void rust_bash_result_free(CExecResult* result);
```

Behavior:
1. If `result` is `NULL`, no-op
2. Reclaim stdout bytes: reconstruct `Box<[u8]>` from `(stdout_ptr, stdout_len)` and drop
3. Reclaim stderr bytes: same
4. Drop the `Box<CExecResult>`

### 2f. Implement `rust_bash_free`

```c
// C signature:
void rust_bash_free(RustBash* sb);
```

Behavior:
1. If `sb` is `NULL`, no-op
2. Reconstruct `Box<RustBash>` from raw pointer and drop

### 2g. Implement `rust_bash_version`

```c
// C signature:
const char* rust_bash_version(void);
```

Returns a pointer to a static null-terminated version string (`env!("CARGO_PKG_VERSION")`). Valid for the lifetime of the library. Useful for runtime version checks and debugging.

### 2h. Integration tests

Create `tests/ffi.rs` with tests that call the FFI functions directly from Rust (using `unsafe`). The file must start with `#![cfg(feature = "ffi")]` so tests are skipped when the feature is off:

**Happy path:**
- Create with `NULL` config → exec "echo hello" → stdout = "hello\n", exit_code = 0 → free
- Create with JSON files → exec "cat /file" → correct content
- Create with JSON env → exec "echo $VAR" → correct value
- Create with JSON cwd → exec "pwd" → correct path
- State persistence: exec "X=42" → exec "echo $X" → "42\n"
- Multiple exec calls with varying commands

**Error handling:**
- Exec with `NULL` sandbox pointer → returns `NULL`, last_error = "Null sandbox pointer"
- Exec with `NULL` command pointer → returns `NULL`, last_error set
- Create with invalid JSON → returns `NULL`, last_error contains parse error
- Create with valid JSON but invalid config → returns `NULL`, last_error set
- Last_error is `NULL` after a successful call (cleared)

**Edge cases:**
- `rust_bash_free(NULL)` → no-op (no crash)
- `rust_bash_result_free(NULL)` → no-op (no crash)
- Create with empty JSON `{}` → valid default sandbox
- Exec with empty command `""` → valid result, exit_code = 0
- Version returns non-null string

**Limits & network:**
- Create with `max_command_count: 1` → exec `"echo a; echo b"` (two sub-commands in one exec) → returns `NULL`, last_error contains "LimitExceeded". Note: execution counters reset at the start of each `exec()` call, so limits are per-exec, not cumulative across calls.
- Create with `max_execution_time_secs: 1` → exec a long-running command → returns `NULL`, last_error contains timeout info
- Create with network policy → curl respects URL allow-list

**Empty string / edge cases:**
- Exec a command that produces no output (e.g., `"true"`) → `stdout_len == 0`, verify `result_free` handles len==0 correctly without crashing
- Exec a command that produces only stderr → `stdout_len == 0`, `stderr_len > 0`
- Unicode output: exec `"echo '日本語'"` → verify stdout bytes are correct UTF-8

### 2i. Documentation

- Add doc comments with `# Safety` sections to all `pub extern "C"` functions
- Document memory ownership rules in module-level docs
- Document thread safety constraints: "A `RustBash*` handle must not be shared across threads without external synchronization"

**Exit criteria**: All 6 FFI functions work correctly. `cargo test --features ffi` passes all integration tests. `cargo build --features ffi --release` produces a shared library with correct exported symbols.

## Phase 3 — C Header Generation ✅

> Generate `include/rust_bash.h` using cbindgen and verify it's valid C.

### 3a. Install cbindgen

```bash
cargo install cbindgen
```

cbindgen is used as a CLI tool (not a build dependency) to keep the build simple. The generated header is version-controlled so downstream consumers don't need cbindgen.

### 3b. Create `cbindgen.toml`

```toml
language = "C"
include_guard = "RUST_BASH_H"
tab_width = 4
documentation_style = "c99"
style = "both"
cpp_compat = true

header = "/* Auto-generated by cbindgen. Do not edit manually. */"

[export]
include = ["CExecResult"]

[export.rename]
"RustBash" = "RustBash"
"CExecResult" = "ExecResult"
```

The rename of `CExecResult` → `ExecResult` in the generated header aligns with the Chapter 8 spec (which uses `ExecResult`), while Rust code uses `CExecResult` to avoid collision with the existing `ExecResult` type.

### 3c. Generate header

```bash
cbindgen --config cbindgen.toml --crate rust-bash --output include/rust_bash.h --features ffi
```

The generated header should contain:
- `typedef struct RustBash RustBash;` (opaque type)
- `typedef struct CExecResult { ... } CExecResult;`
- Function declarations for all 6 FFI functions
- Include guard `#ifndef RUST_BASH_H`

### 3d. Verify header compiles

```bash
mkdir -p include
cc -fsyntax-only include/rust_bash.h    # GCC/Clang syntax check
```

### 3e. Add header regeneration note

Add a comment at the top of the generated header:

```c
/* Auto-generated by cbindgen. Do not edit manually.
 * Regenerate with: cbindgen --config cbindgen.toml --crate rust-bash --output include/rust_bash.h --features ffi
 */
```

### 3f. Documentation

- Document the header generation command in `README.md` (FFI section)
- Mention cbindgen version used

**Exit criteria**: `include/rust_bash.h` exists, is valid C, and declares all FFI functions and types. Header is version-controlled.

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Panic unwinding across FFI boundary → UB | Medium | Critical | Wrap every `extern "C"` fn in `catch_unwind`; convert panics to `set_last_error` + NULL return |
