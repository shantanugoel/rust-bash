# Milestones 5.3 & 5.4 — WASM Target, npm Package, AI SDK Integration & Showcase Website

## Problem Statement

rust-bash has a production-quality interpreter (M1–M4) with a CLI binary (M5.1) and C FFI (M5.2), but no JavaScript/TypeScript integration. To compete with [just-bash](https://github.com/vercel-labs/just-bash) — the TypeScript bash interpreter that inspired this project — we need:

1. **WASM compilation** so rust-bash runs in browsers and edge runtimes
2. **An npm package** with a TypeScript API at least as ergonomic as just-bash's `Bash` class
3. **Native Node.js addon** via napi-rs for maximum server-side performance
4. **Custom command callbacks** from TypeScript (equivalent to just-bash's `defineCommand`)
5. **AI SDK integration** — framework-agnostic tool definitions for OpenAI / Anthropic / any LLM, plus MCP server mode
6. **A showcase website** (like [justbash.dev](https://justbash.dev)) with a live interactive terminal

### Design Decision: napi-rs v3 Unified Build

After evaluating the options from the guidebook's design exploration notes:

- **Chosen approach**: Separate `wasm-bindgen` (WASM) + `napi-rs` (native Node.js addon) builds, unified under a single `rust-bash` npm package that auto-detects the environment.
- **Rationale**: napi-rs v3's WASM support is still maturing and may constrain the API surface. Separate binding layers give full control over each target while the TypeScript wrapper provides a unified developer experience.
- **Package structure**: Single `rust-bash` package with conditional exports — native addon for Node.js, WASM for browsers/edge. No need for users to choose between packages.

### Competitive Positioning vs just-bash

| Feature | just-bash | rust-bash (target) |
|---|---|---|
| Language | Pure TypeScript | Rust → WASM + native addon |
| Performance | JS-speed | Near-native (native addon) / WASM-fast (browser) |
| Bundle size | ~2MB+ (JS) | ~1–1.5MB gzipped (WASM, needs validation) |
| API | `new Bash(opts)` + `exec()` | `new Bash(opts)` + `exec()` (same API shape) |
| Custom commands | `defineCommand(name, fn)` | `defineCommand(name, fn)` (same pattern) |
| AI tool integration | Vercel AI SDK only (`bash-tool`) | Framework-agnostic: JSON Schema + handler; MCP server; recipe adapters |
| MCP server | ❌ | ✅ (`rust-bash --mcp`) |
| Python integration | ❌ | ✅ (C FFI / future PyPI package) |
| Browser | ✅ | ✅ (WASM) |
| Node.js native | ❌ (JS only) | ✅ (napi-rs addon) |

## Phase 1: WASM Compilation Foundation ✅

**Goal**: Get rust-bash compiling to `wasm32-unknown-unknown` and callable from JavaScript.

### 1.1 — WASM Feature Gates and Platform Abstraction

Add conditional compilation for WASM-incompatible code. This is the most critical step — several stdlib types don't work on `wasm32-unknown-unknown` and their usage is pervasive.

#### Time abstraction (`web-time` migration) — **high effort**

`SystemTime` and `Instant` are used throughout the codebase and **panic on `wasm32-unknown-unknown`**. The `web-time` crate provides drop-in replacements that use `js_sys::Date` on WASM and delegate to `std::time` on native.

Create `src/platform.rs` with re-exports:
```rust
#[cfg(target_arch = "wasm32")]
pub use web_time::{SystemTime, Instant, UNIX_EPOCH};
#[cfg(not(target_arch = "wasm32"))]
pub use std::time::{SystemTime, Instant, UNIX_EPOCH};
```

Then replace all `std::time::{SystemTime, Instant}` imports with `crate::platform::*`. Known locations (**30+ call sites**):
- **VFS layer**: `SystemTime` stored in `Metadata`, `FsNode::File`, `FsNode::Directory`, `FsNode::Symlink`, and the `VirtualFs::utimes()` trait signature — across `memory.rs`, `overlay.rs`, `mountable.rs`, `readwrite.rs`, and the trait definition in `mod.rs`
- **Interpreter**: `Instant::now()` in `ExecutionCounters::default()` — called on every `exec()` invocation (hot path)
- **Commands**: `awk` `srand()` in `src/commands/awk/runtime.rs` uses `SystemTime::now()` directly
- **date command**: `chrono::Local::now()` in `src/commands/utils.rs` — needs `chrono = { features = ["wasmbind"] }` for WASM

#### `std::thread::sleep` — incompatible with WASM

`src/commands/utils.rs` calls `std::thread::sleep(duration)` which will **panic or hang** on WASM. Feature-gate the `sleep` command:
- On native: works as-is
- On WASM: return an error `"sleep: not supported in browser environment"` (matches just-bash which also doesn't support real sleep in browser)

#### `chrono` WASM compatibility

Add `wasmbind` feature for WASM builds so `chrono::Local::now()` works. Without this, it falls back to UTC with no timezone info.

#### Networking

- **`ureq` / `url`**: Feature-gate behind `network` cargo feature. Currently these are unconditional deps — extract them as optional. **Keep `network` in the default feature set** to avoid breaking existing native builds.
- The `curl` command returns "command not found" when network feature is unavailable (matches just-bash behavior).

#### Filesystem backends

- **OverlayFs / ReadWriteFs**: Feature-gate behind `native-fs` feature (these use `std::fs`). WASM builds only get `InMemoryFs` and `MountableFs`.
- Conditional `pub use` in `lib.rs` and `#[cfg]` on module declarations.
- Note: `MountableFs` may reference `OverlayFs`/`ReadWriteFs` in some code paths — trace the dependency graph and use `#[cfg]` appropriately.

#### Other items

- **`rustyline` / `clap`**: Already feature-gated behind `cli` — no action needed.
- **Thread-local storage in FFI**: Not needed for WASM — feature-gate the `ffi` module.
- **`parking_lot`**: Compiles to WASM (falls back to spin-locks since WASM is single-threaded). Smoke-test for re-entrancy issues (e.g., custom command that calls `exec()` while holding a lock).
- **`serde_json`**: Already an unconditional dependency (used by jq) — compiles to WASM fine, just adds to bundle size.

Cargo.toml changes:
```toml
[features]
default = ["cli", "network"]
cli = ["dep:clap", "dep:rustyline"]
ffi = ["dep:serde"]
wasm = ["dep:wasm-bindgen", "dep:js-sys", "dep:web-time", "dep:serde", "dep:serde-wasm-bindgen"]
network = ["dep:ureq", "dep:url"]
native-fs = []  # OverlayFs, ReadWriteFs — on by default for native

[target.'cfg(not(target_arch = "wasm32"))'.dependencies]
ureq = { version = "3.0", optional = true }
url = { version = "2.5.8", optional = true }

[target.'cfg(target_arch = "wasm32")'.dependencies]
wasm-bindgen = { version = "0.2", optional = true }
js-sys = { version = "0.3", optional = true }
web-time = { version = "1.1", optional = true }
serde-wasm-bindgen = { version = "0.6", optional = true }
chrono = { version = "0.4", features = ["wasmbind"] }
```

**Critical first step**: Do a smoke build (`cargo build --target wasm32-unknown-unknown --features wasm --no-default-features`) **before writing any wasm-bindgen code** to flush out all compilation failures. Track actual binary size from this first build.

**Tests**: The smoke build compiles cleanly. All existing `cargo test` (native) still pass after the refactor.

### 1.2 — wasm-bindgen Bindings

Create `src/wasm.rs` (feature-gated behind `wasm`):

```rust
#[wasm_bindgen]
pub struct WasmBash { inner: RustBash }

#[wasm_bindgen]
impl WasmBash {
    #[wasm_bindgen(constructor)]
    pub fn new(config: JsValue) -> Result<WasmBash, JsError>;
    
    pub fn exec(&mut self, command: &str) -> Result<JsValue, JsError>;
    pub fn exec_with_options(&mut self, command: &str, options: JsValue) -> Result<JsValue, JsError>;
    
    // Filesystem access
    pub fn write_file(&mut self, path: &str, content: &str) -> Result<(), JsError>;
    pub fn read_file(&self, path: &str) -> Result<String, JsError>;
    pub fn mkdir(&mut self, path: &str, recursive: bool) -> Result<(), JsError>;
    
    // State
    pub fn cwd(&self) -> String;
    pub fn last_exit_code(&self) -> i32;
    pub fn command_names(&self) -> Vec<String>;
    
    // Custom command registration (via JS callback)
    pub fn register_command(&mut self, name: &str, callback: js_sys::Function) -> Result<(), JsError>;
}
```

The `config` JsValue accepts the same JSON shape as the C FFI config, plus:
- `files`: `Record<string, string>` — seed VFS
- `env`: `Record<string, string>` — environment variables
- `cwd`: `string` — working directory
- `executionLimits`: `Partial<ExecutionLimits>` — limits config

`exec_with_options` accepts:
- `env`: per-exec environment overrides
- `cwd`: per-exec working directory
- `stdin`: standard input string

> **Note on `AbortSignal` cancellation**: The current interpreter has no cooperative cancellation mechanism — the closest is `max_execution_time` checked via `Instant::elapsed()`. True `AbortSignal` support requires adding `Arc<AtomicBool>` to `InterpreterState` and checking it at command dispatch points (see M9.1 — Cooperative Cancellation). **Deferred to a future phase** — the execution time limit already provides safety for runaway scripts.

### 1.3 — Custom Command Callbacks (WASM → JS bridge)

Implement `JsBridgeCommand` — a Rust struct that implements `VirtualCommand` but delegates execution to a JavaScript callback:

```rust
struct JsBridgeCommand {
    name: String,
    callback: js_sys::Function,
}

impl VirtualCommand for JsBridgeCommand {
    fn name(&self) -> &str { &self.name }
    fn execute(&self, args: &[String], ctx: &mut CommandContext) -> Result<ExecOutput, CommandError> {
        // Serialize args + context to JsValue
        // Call callback.call1(&JsValue::NULL, &js_args)
        // Deserialize result { stdout, stderr, exitCode }
    }
}
```

> **⚠️ Sync/async constraint**: The Rust `VirtualCommand::execute()` trait is **synchronous**. In WASM, you cannot block the main thread waiting for a JS `Promise` to resolve. Therefore:
> - **WASM custom commands must be synchronous** — the callback must return `{ stdout, stderr, exitCode }` directly, NOT a `Promise`.
> - **napi-rs native addon custom commands can be async** — `ThreadsafeFunction` allows blocking on the Rust side while the JS callback completes asynchronously.
> - The TypeScript `defineCommand()` API should accept both sync and async signatures, but document that async commands only work with the native addon backend, not in browser WASM.

The JS callback receives `(args: string[], ctx: CommandContext)` where `CommandContext` includes:
- `fs`: proxy object with `readFileSync(path)`, `writeFileSync(path, content)`, `readdirSync(path)`, `statSync(path)`, `existsSync(path)`, `mkdirSync(path, opts)`, `rmSync(path, opts)` — mapped to VFS operations
- `cwd`: current working directory
- `env`: environment variables as `Record<string, string>`
- `stdin`: standard input content
- `exec(command: string)`: callback to execute sub-commands (enables `xargs`-like custom commands)

This matches just-bash's `defineCommand` context interface, making migration trivial.

The TypeScript API:
```typescript
import { Bash, defineCommand } from 'rust-bash';

const hello = defineCommand("hello", async (args, ctx) => {
  const name = args[0] || "world";
  return { stdout: `Hello, ${name}!\n`, stderr: "", exitCode: 0 };
});

const bash = new Bash({ customCommands: [hello] });
await bash.exec("hello Alice"); // "Hello, Alice!\n"
```

### 2.2 — TypeScript Bash Class (API Design)

The core `Bash` class wraps either the native addon or WASM module:

```typescript
export interface BashOptions {
  files?: Record<string, string | (() => string) | (() => Promise<string>)>;
  env?: Record<string, string>;
  cwd?: string;
  executionLimits?: Partial<ExecutionLimits>;
  customCommands?: CustomCommand[];
  // Features not in just-bash (our advantages):
  network?: NetworkConfig;    // URL allow-list (just-bash also has this)
}

export interface ExecOptions {
  env?: Record<string, string>;
  replaceEnv?: boolean;
  cwd?: string;
  stdin?: string;
  signal?: AbortSignal;
  rawScript?: boolean;
  args?: string[];
}

export interface ExecResult {
  stdout: string;
  stderr: string;
  exitCode: number;
  env?: Record<string, string>;
}

export class Bash {
  readonly fs: FileSystemProxy;  // Read/write VFS from JS

  constructor(options?: BashOptions);
  exec(command: string, options?: ExecOptions): Promise<ExecResult>;
  
  // Convenience methods (not in just-bash)
  writeFile(path: string, content: string): void;
  readFile(path: string): string;
  getCwd(): string;
  getCommandNames(): string[];
  
  // Transform plugins (like just-bash's AST transform API)
  registerTransformPlugin(plugin: TransformPlugin): void;
}
```

Key design decisions vs just-bash:
- **Same constructor shape**: `new Bash({ files, env, cwd })` — drop-in replacement
- **Same exec API**: `bash.exec(cmd, opts)` returns `{ stdout, stderr, exitCode }`
- **Lazy files support**: `files` values can be `() => string` or `() => Promise<string>` (matches just-bash)
- **`exec()` is async**: Even though the Rust core is sync, wrapping in `Promise` gives consistent API and allows for future async features
- **fs proxy**: Expose VFS operations directly (matches just-bash's `bash.fs`)

**Tests**: API compatibility tests ensuring the same scripts produce the same output as just-bash.

### 2.3 — Lazy File Loading

Implement lazy file materialization support. When `files` values are functions:
- Store a placeholder in the VFS
- On first `read_file`, call the function, cache the result, then return it
- If the file is written before being read, the lazy callback is never invoked

This matches just-bash's lazy file API:
```typescript
const bash = new Bash({
  files: {
    "/data/config.json": '{"key": "value"}',                           // eager
    "/data/large.csv": () => "col1,col2\na,b\n",                       // lazy sync
    "/data/remote.txt": async () => (await fetch("...")).text(),        // lazy async
  },
});
```

Implementation: The TS wrapper resolves lazy files before passing to the WASM/native core, with a proxy layer that intercepts VFS reads.

### 2.6 — High-Level Convenience API

Add convenience features to the `Bash` class that enrich the developer experience beyond the basic `exec()` API. These are features found in mature sandbox runtimes, integrated directly into our own API rather than mimicking any vendor's interface:

- **Command filtering**: `commands` option restricts which commands are available per-instance. Critical for least-privilege sandboxing.
- **Per-exec env/cwd isolation**: `exec()` accepts `env`, `cwd`, `replaceEnv` overrides that are scoped to that execution.
- **Safe argument passing**: `args` option for additional argv entries that bypass shell parsing (no escaping/splitting/globbing).
- **Script normalization**: Strip leading whitespace from template literals while preserving heredoc content. `rawScript: boolean` option to disable.

```typescript
const bash = new Bash({
  commands: ['echo', 'cat', 'grep', 'jq'],  // allow-list
  files: { '/data.json': '{"key": "value"}' },
});

const result = await bash.exec('jq .key /data.json', {
  env: { LANG: 'en_US.UTF-8' },
  cwd: '/data',
});
```

## Phase 3: AI SDK Integration

**Goal**: Framework-agnostic AI tool primitives, MCP server mode, and documented recipe adapters for popular frameworks.

### 3.1 — Framework-Agnostic Tool Primitives

Export universal building blocks from `rust-bash` that work with **any** AI agent framework. The core exports a JSON Schema tool definition and a handler factory — no framework dependencies.

```typescript
// rust-bash — the universal primitive
import { bashToolDefinition, createBashToolHandler } from 'rust-bash';

// bashToolDefinition is a plain JSON Schema object:
// {
//   name: 'bash',
//   description: 'Execute bash commands in a sandboxed environment...',
//   inputSchema: {
//     type: 'object',
//     properties: { command: { type: 'string', description: '...' } },
//     required: ['command'],
//   },
// }

// createBashToolHandler returns a framework-agnostic handler:
const { handler, definition, bash } = createBashToolHandler({
  files: { '/data.txt': 'hello world' },
  maxOutputLength: 10000,
});

// handler: (args: { command: string }) => Promise<{ stdout, stderr, exitCode }>
const result = await handler({ command: 'grep hello /data.txt' });
```

#### Usage with any framework (recipe examples)

**OpenAI API (direct — no adapter needed):**

```typescript
// In tool call dispatch:
const toolArgs = JSON.parse(toolCall.function.arguments);

```


```typescript

```

### 3.3 — MCP Server Mode

Add `--mcp` flag to the CLI binary (`rust-bash --mcp`). This launches an MCP (Model Context Protocol) server over stdio using JSON-RPC, making rust-bash instantly available to any MCP-compatible client — Claude Desktop, ChatGPT, Cursor, VS Code, Windsurf, Cline, and the OpenAI Agents SDK (`HostedMCPTool`).

**Exposed MCP tools:**
- `bash` — execute commands (accepts `{ command: string }`)
- `write_file` — write content to VFS (accepts `{ path, content }`)
- `read_file` — read file from VFS (accepts `{ path }`)
- `list_directory` — list directory contents (accepts `{ path }`)

**Configuration (in any MCP client):**

**Replay with typewriter animation** — same rendering code as live agent, but with timing:

```typescript
export async function* replayCache(
  events: AgentEvent[],
  bash?: Bash,
): AsyncGenerator<AgentEvent> {
  let interrupted = false;
  const onInterrupt = () => { interrupted = true; };

  for (const event of events) {
    if (interrupted) { yield event; continue; } // Skip animation, dump rest instantly

    if (event.type === 'text') {
      // Simulate LLM streaming: ~15ms per character with slight irregularity
      for (const char of event.content) {
        if (interrupted) { yield { type: 'text', content: event.content.slice(event.content.indexOf(char)) }; break; }
        yield { type: 'text', content: char };
        await sleep(12 + Math.random() * 8);
      }
    } else if (event.type === 'tool_call') {
      await sleep(100);
      // Execute the command for real so VFS state is consistent
      if (bash) bash.exec(event.command);
      yield event; // Tool calls render as a block (instant)
      await sleep(300);
    }
  }

  return onInterrupt; // Caller wires this to term.onKey
}
```

**Key design decisions**:
- **Hand-written, not LLM-generated**: The initial demo is marketing — it should be perfect and deterministic. No API call needed. Commit it to the repo.
- **Same `AgentEvent[]` type as live agent**: Terminal rendering has zero branching between cached and live responses.
- **VFS execution during replay**: Tool calls in the cached response are executed for real against WASM, so `ls` shows the same files the agent mentioned.
- **Interruption**: If the user types during animation, skip remaining animation and dump output instantly. Don't cancel — they might want to see the rest.
- **WASM race condition**: The typing animation (~2.3s) buys time for WASM to load. The cached text animation can even start before WASM is ready — only tool call VFS execution needs WASM, and those are queued in the background.

The "is this the matrix?" command ties perfectly into the matrix rain loading screen — users see the rain, then the terminal emerges, then the first thing the agent says is answering that exact question. Full thematic arc.

- `agent "your question"` — explicit invocation, not a mode switch. It's just a command.
- No auto-detection of natural language. `find all python files` should never be ambiguous — in a terminal, what you type is what runs.
- `agent` with no args → shows usage and examples.
- The agent's tool calls modify the shared Bash/VFS instance, so after `agent "create a fibonacci script"`, the user can `cat fib.sh` and see it.
- Up arrow recalls previous agent queries too (they're in the same history).

**Tool call rendering** — the critical UX pattern (matching justbash.dev style):

Rendering rules:
1. **LLM streaming text** → render inline as it arrives
2. **Tool call** → render indented: `  $ command` (dim `$`, cyan command) + indented output

### 4.3 — AI Agent (Client-Side Loop + Cloudflare Worker Proxy)

The agent architecture splits into two parts: a **Cloudflare Worker** that proxies LLM requests (hiding the API key), and a **client-side agent loop** that handles tool call detection and local WASM execution.

#### Cloudflare Worker (LLM Proxy — `functions/api/chat.ts`)

A minimal ~30-line Cloudflare Pages Function that proxies requests to Google Gemini 3.1 Flash Lite Preview. The API key is stored as a Worker secret, never exposed to the client.

```typescript

  const ip = request.headers.get('CF-Connecting-IP');
  // ... rate limiting via KV or in-memory counter ...
```

#### Client-Side Agent Loop (`src/agent.ts`)

The agent loop runs entirely in the browser. It is a pure async generator that yields events (`text` and `tool_call`). The terminal layer (§4.2) consumes these events for rendering — the agent module has no terminal/UI dependencies.

Tool calls are executed locally via the already-loaded rust-bash WASM — no server roundtrip needed for bash execution.

```typescript
// src/agent.ts — pure agent loop, yields events for the terminal to render
import OpenAI from 'openai';
import type { Bash } from 'rust-bash/browser';
import { bashToolDefinition } from 'rust-bash';

export type AgentEvent =
  | { type: 'text'; content: string }
  | { type: 'tool_call'; command: string; result: ExecResult };

const client = new OpenAI({
  baseURL: '/api',  // proxied through CF Worker
  apiKey: 'unused', // Worker handles auth
  dangerouslyAllowBrowser: true,
});

const tools = [{
  type: 'function' as const,
  function: {
    name: bashToolDefinition.name,
    description: bashToolDefinition.description,
    parameters: bashToolDefinition.inputSchema,
  },
}];

export async function* runAgentLoop(
  userMessage: string,
  bash: Bash,
): AsyncGenerator<AgentEvent> {
  const messages = [
    { role: 'system' as const, content: SYSTEM_INSTRUCTIONS },
    { role: 'user' as const, content: userMessage },
  ];

  // Multi-turn loop: LLM may request multiple tool calls in sequence
  while (true) {
    const stream = await client.chat.completions.create({
      model: 'gemini-3.1-flash-lite-preview',
      messages,
      tools,
      stream: true,
    });

    let fullResponse = '';
    let toolCalls: any[] = [];

    for await (const chunk of stream) {
      const delta = chunk.choices[0]?.delta;
      if (delta?.content) {
        fullResponse += delta.content;
        yield { type: 'text', content: delta.content };
      }
      if (delta?.tool_calls) {
        for (const tc of delta.tool_calls) {
          toolCalls[tc.index] = toolCalls[tc.index] || { id: '', function: { name: '', arguments: '' } };
          if (tc.id) toolCalls[tc.index].id = tc.id;
          if (tc.function?.name) toolCalls[tc.index].function.name = tc.function.name;
          if (tc.function?.arguments) toolCalls[tc.index].function.arguments += tc.function.arguments;
        }
      }
    }

    if (toolCalls.length > 0) {
      messages.push({ role: 'assistant' as const, content: fullResponse || null, tool_calls: toolCalls });

      // Execute each tool call locally via WASM — instant, no server roundtrip
      for (const tc of toolCalls) {
        const args = JSON.parse(tc.function.arguments);
        const result = bash.exec(args.command);
        yield { type: 'tool_call', command: args.command, result };

        messages.push({
          role: 'tool' as const,
          tool_call_id: tc.id,
          content: JSON.stringify({
            stdout: result.stdout.slice(0, 5000),
            stderr: result.stderr.slice(0, 2000),
            exitCode: result.exitCode,
          }),
        });
      }
      // Loop continues — send results back to LLM for next turn
    } else {
      // No tool calls — final text response, done
      break;
    }
  }
}
```

The terminal layer in §4.2 calls `runAgentLoop()` and renders each event:
- `text` events → streamed to terminal inline
- `tool_call` events → rendered as indented `$ command` + output (see §4.2 rendering rules)

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| WASM binary too large (> 2MB gzipped) | Medium | High | Feature-gate heavy deps (`jaq`, `regex`); use `wasm-opt -Oz`; do a smoke build early in §1.1 to get actual size before committing to marketing claims |
| `brush-parser` WASM compat issues | Low | High | Parser already compiles to wasm32; test early in Phase 1.1 — check transitive dep tree |
| napi-rs cross-platform build failures | Medium | Medium | Use napi-rs's battle-tested CI templates; pre-build for 6 major platform targets |
| Custom command callback performance (WASM↔JS) | Medium | Medium | Batch context serialization; cache `CommandContext` proxy objects |
| `SystemTime`/`Instant` WASM migration scope | High | Medium | 30+ call sites across VFS — `web-time` crate is drop-in but the refactor touches every VFS file; budget for this |
| Provider API churn | Low | Low | Tool definitions use JSON Schema (stable standard); `openai` npm package is the de facto stable interface; no framework-specific dependencies in core |
| Vite WASM loading | Low | Low | Vite handles WASM via `?url` or `vite-plugin-wasm`; simpler than Next.js webpack config |
| Gemini free tier changes/removal | Medium | High | Worker supports multi-provider fallback (Groq, Workers AI); switch is a single env var change |
| Gemini free tier rate limit under traffic | Medium | Medium | Per-IP rate limiting in Worker (10 req/min); auto-fallback to Workers AI/Groq on 429; friendly "demo busy" message |
| Gemini free tier data usage for training | Low | Low | Acceptable for a public demo; no sensitive data is involved |
| MCP protocol evolution | Low | Low | MCP spec is versioned; implement core subset (tools/list, tools/call) which is stable |
| `parking_lot` spin-lock re-entrancy on WASM | Low | High | Single-threaded WASM may hit deadlocks if `RwLock` is held across re-entrant calls (custom command → `exec()` → WASM → JS → WASM). Add depth guard |
| WASM custom command re-entrancy stack overflow | Medium | High | Custom command calls `exec()` which calls back into WASM → JS → WASM. Works but can hit WASM stack limits. Enforce a depth guard (e.g., max 5 levels) |
| npm scope `@rust-bash` availability | Low | High | Verify npm org availability before building around this package name |
| `serde-wasm-bindgen` / `wasm-bindgen` version alignment | Medium | Medium | These must be version-aligned; pin carefully in Cargo.toml |

## Success Criteria

1. **`npm install rust-bash`** works and provides a TypeScript API matching just-bash's `Bash` class
2. **`new Bash({ files, env, cwd }).exec(cmd)`** works in both Node.js and browsers
3. **`defineCommand()`** allows TypeScript custom commands with full context (fs, env, exec)
4. **Framework-agnostic tool definitions** work with OpenAI, Anthropic, Vercel AI SDK, LangChain, and any other framework. **MCP server mode** works with Claude Desktop, Cursor, and VS Code.
5. **Showcase website** at `examples/website/` runs rust-bash WASM in the browser with an AI agent, deployed on Cloudflare Pages ($0/month), using Gemini 3.1 Flash Lite Preview with Workers AI/Groq fallback
6. **Performance**: Native addon is faster than just-bash; WASM is comparable
7. **Bundle size**: WASM < 2MB gzipped (target 1.5MB; validate with smoke build in §1.1)
8. **Zero breaking changes** to existing Rust crate API, CLI, or C FFI
