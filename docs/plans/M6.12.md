# Milestone 6.12: Differential Testing Against Real Bash — Implementation Plan

## Problem Statement

rust-bash has 4,700+ lines of integration tests, but all expected values are hand-written. There is no automated way to verify that rust-bash's behavior matches real `/bin/bash`. Word expansion edge cases and array behaviors are explicitly called out in the risk register as needing differential testing. M6.12 builds a fixture-based comparison test suite that:

1. Records expected output from real `/bin/bash` once (fixture files).
2. Runs those same scripts through rust-bash on every `cargo test`.
3. Compares stdout, stderr, and exit code — surfacing regressions and compatibility gaps.
4. Provides dedicated spec-test suites for `awk`, `grep`, `sed`, and `jq` mini-languages.

## Prerequisites & Carryover

M6.12 is independent in the dependency graph — it can start before other M6 milestones. However, the comparison test suite becomes more valuable as more M6 features land. Recommended approach:

- **Start with Phase 1 (infrastructure) immediately** — the runner is useful from day one.
- **Phase 2 fixtures should test currently-implemented features first.** As M6.1 (arrays), M6.3 (shopt), M6.11 (parameter transforms), etc. land, add comparison tests for those features. Use `skip` for tests that exercise unimplemented features.
- **Phase 3 spec tests** can proceed immediately since awk/grep/sed/jq are already implemented.

The existing `tests/integration.rs` (4,700+ lines, ~460 tests) covers many of the same shell features but with hand-written expected values. The comparison suite is **complementary, not a replacement**: integration tests validate the Rust API ergonomics and are fast to write; comparison fixtures validate bash compatibility via recorded ground truth. Avoid duplicating trivial tests like `echo hello` — focus comparison tests on edge cases and behaviors known to diverge from real bash.

As commands are added in M7, the comparison suite should expand to cover command output format matching (e.g., `ls`, `stat`, `date` output formats). This is future scope, not part of M6.12's initial delivery.

## Design Decisions

### 1. Fixture format: TOML test manifests

Each test domain gets a directory of `.toml` fixture files. TOML is chosen over raw text or JSON because it handles multi-line strings cleanly, is already parseable in Rust, and is self-documenting.

```toml
# tests/fixtures/comparison/expansion/parameter_default.toml

[[cases]]
name = "default_value_unset"
script = 'echo ${UNSET:-fallback}'
stdout = "fallback\n"
stderr = ""
exit_code = 0

[[cases]]
name = "default_value_set"
script = '''
X=hello
echo ${X:-fallback}
'''
stdout = "hello\n"
stderr = ""
exit_code = 0
```

The `stdout`, `stderr`, and `exit_code` fields are the **recorded fixtures** — output from real bash. When `RECORD_FIXTURES=1` is set, the runner overwrites these fields by executing each script against `/bin/bash`.

### 2. Two-mode runner architecture

- **Normal mode** (`cargo test`): Read fixtures, run scripts through `RustBash::exec()`, compare against recorded values. No real bash needed — works in CI, WASM, or any environment.
- **Record mode** (`RECORD_FIXTURES=1 cargo test`): Run each script against real `/bin/bash` via `std::process::Command`, capture stdout/stderr/exit_code, and write updated fixture files back to disk. This is the only code path that uses `std::process::Command` — it lives in test code only, not library code.

### 3. Spec tests vs comparison tests — separate suites

- **Comparison tests** (`tests/fixtures/comparison/`): Shell language features tested against real bash. These validate rust-bash's interpreter.
- **Spec tests** (`tests/fixtures/spec/`): Domain-specific tests for `awk`, `grep`, `sed`, `jq`. These test the command implementations in isolation with known inputs/outputs, not necessarily recorded from real bash (since our implementations are intentionally subset).

### 4. Per-file test discovery via `datatest-stable`

Use `datatest-stable` to discover `.toml` fixture files and generate one `#[test]` per file. This ensures:
- Each fixture file appears as a named test in `cargo test` output.
- Failures in one file don't prevent other files from running.
- Individual files can be run with `cargo test comparison::quoting` etc.
- Cargo's built-in parallelism runs fixture files concurrently.

Within each file, cases run sequentially — a failure in one case reports all failures in that file before moving on (using soft assertions / collecting errors). Adding a new test case is just adding a TOML entry — no Rust code changes needed.

### 5. Environment isolation

All comparison tests run in a clean `RustBash` sandbox with:
- No inherited environment variables (deterministic)
- Controlled timezone (`TZ=UTC`)
- Known `$HOME`, `$USER`, `$PATH` values (set identically in both modes)
- Files seeded only from the fixture's optional `files` field
- Execution limits set explicitly (max 10,000 iterations, 5s wall-clock) to prevent hangs from infinite-loop bugs

The recording mode similarly runs bash with `env -i` plus only the controlled variables, to avoid host-specific leakage. `PATH` is set to `/usr/local/bin:/usr/bin:/bin` on both sides to avoid asymmetry.

### 6. Stderr comparison strategy

Stderr messages from rust-bash may differ in exact wording from real bash (e.g., `bash: foo: command not found` vs `foo: command not found`). The comparison supports three modes per test case:

- `stderr = "exact string"` — exact match (default)
- `stderr_contains = "substring"` — partial match
- `stderr_ignore = true` — skip stderr comparison entirely (useful when we only care about exit code)

## Phase 1 — Test Infrastructure & Runner

Build the fixture format, TOML parsing, and both test runners (comparison + spec).

### 1a. Define the fixture data model

Create shared types for deserializing TOML fixtures:

```rust
// In tests/comparison.rs (or a shared test utility module)

#[derive(serde::Deserialize)]
struct FixtureFile {
    cases: Vec<TestCase>,
}

#[derive(serde::Deserialize)]
struct TestCase {
    name: String,
    script: String,
    stdout: String,
    #[serde(default)]
    stderr: String,
    #[serde(default)]
    exit_code: i32,
    // Flexible stderr comparison
    #[serde(default)]
    stderr_contains: Option<String>,
    #[serde(default)]
    stderr_ignore: bool,
    // Optional stdin content piped to the script
    #[serde(default)]
    stdin: Option<String>,
    // If true, expect exec() to return Err (parse errors, etc.)
    // The test passes if exec() errors and exit_code matches
    #[serde(default)]
    expect_error: bool,
    // Optional VFS files to seed before running
    #[serde(default)]
    files: std::collections::HashMap<String, String>,
    // Optional environment variables
    #[serde(default)]
    env: std::collections::HashMap<String, String>,
    // Optional: skip this test (with reason)
    #[serde(default)]
    skip: Option<String>,
}
```

### 1b. Build the comparison test runner

Use `datatest-stable` to register a test function that receives a file path. The crate discovers all `.toml` files and creates one `#[test]` per file:

```rust
fn run_comparison_file(path: &Path) -> datatest_stable::Result<()> {
    let content = std::fs::read_to_string(path)?;
    let fixture: FixtureFile = toml::from_str(&content)?;
    let mut failures = Vec::new();

    for case in &fixture.cases {
        if let Some(reason) = &case.skip {
            eprintln!("SKIP {}: {}", case.name, reason);
            continue;
        }

        let mut env_map = HashMap::new();
        env_map.insert("HOME".into(), "/root".into());
        env_map.insert("USER".into(), "testuser".into());
        env_map.insert("TZ".into(), "UTC".into());
        env_map.insert("PATH".into(), "/usr/local/bin:/usr/bin:/bin".into());
        env_map.extend(case.env.clone());

        let mut builder = RustBashBuilder::new().env(env_map);
        // Seed VFS files from the fixture
        if !case.files.is_empty() {
            let file_map: HashMap<String, Vec<u8>> = case.files.iter()
                .map(|(k, v)| (k.clone(), v.as_bytes().to_vec()))
                .collect();
            builder = builder.files(file_map);
        }

        let mut sh = builder.build().unwrap();

        // Handle both Ok and Err results from exec()
        match sh.exec(&case.script) {
            Ok(r) => {
                if r.stdout != case.stdout {
                    failures.push(format!("[{}] STDOUT: expected {:?}, got {:?}",
                        case.name, case.stdout, r.stdout));
                }
                if r.exit_code != case.exit_code {
                    failures.push(format!("[{}] EXIT CODE: expected {}, got {}",
                        case.name, case.exit_code, r.exit_code));
                }
                // Stderr comparison (flexible)
                if !case.stderr_ignore {
                    if let Some(substring) = &case.stderr_contains {
                        if !r.stderr.contains(substring) {
                            failures.push(format!("[{}] STDERR doesn't contain {:?}, got {:?}",
                                case.name, substring, r.stderr));
                        }
                    } else if r.stderr != case.stderr {
                        failures.push(format!("[{}] STDERR: expected {:?}, got {:?}",
                            case.name, case.stderr, r.stderr));
                    }
                }
            }
            Err(e) => {
                if case.expect_error {
                    // Parse error or other exec failure — check exit code convention
                    // (exit code 2 for syntax errors, matching bash)
                } else {
                    failures.push(format!("[{}] exec() returned Err: {}", case.name, e));
                }
            }
        }
    }

    if !failures.is_empty() {
        return Err(format!("{} failure(s) in {}:\n{}",
            failures.len(), path.display(), failures.join("\n")).into());
    }
    Ok(())
}

datatest_stable::harness!(
    run_comparison_file,
    "tests/fixtures/comparison",
    r".*\.toml$",
);
```

### 1c. Build the recording mode

When `RECORD_FIXTURES=1` is set, run each script against real `/bin/bash` and update the TOML files using `toml_edit` for round-trip preservation:

```rust
fn record_from_real_bash(
    script: &str,
    env: &HashMap<String, String>,
    files: &HashMap<String, String>,
) -> (String, String, i32) {
    use std::process::Command;

    // Stage VFS files into a real temp directory for bash to access
    let tmp = tempfile::tempdir().expect("Failed to create temp dir");
    for (path, content) in files {
        // Strip leading / to make relative to temp dir
        let rel_path = path.strip_prefix('/').unwrap_or(path);
        let full_path = tmp.path().join(rel_path);
        if let Some(parent) = full_path.parent() {
            std::fs::create_dir_all(parent).unwrap();
        }
        std::fs::write(&full_path, content).unwrap();
    }

    let mut cmd = Command::new("/bin/bash");
    cmd.arg("-c").arg(script);
    cmd.current_dir(tmp.path());
    cmd.env_clear();
    cmd.env("HOME", "/root");
    cmd.env("USER", "testuser");
    cmd.env("TZ", "UTC");
    cmd.env("PATH", "/usr/local/bin:/usr/bin:/bin");
    for (k, v) in env {
        cmd.env(k, v);
    }
    let output = cmd.output().expect("Failed to execute /bin/bash");
    (
        String::from_utf8_lossy(&output.stdout).to_string(),
        String::from_utf8_lossy(&output.stderr).to_string(),
        output.status.code().unwrap_or(-1),
    )
}
```

The recorder reads each TOML file with `toml_edit`, runs every case against bash, updates the `stdout`/`stderr`/`exit_code` values in-place, and writes the file back. Using `toml_edit` preserves comments, key ordering, and string formatting — avoiding noisy diffs on re-record.

**Important limitation**: Tests with `files` entries use absolute VFS paths (e.g., `/tmp/test/a.txt`). In recording mode, these are staged relative to a temp directory. Scripts that reference absolute paths will see different paths than in the VFS. For these cases, prefer relative paths in scripts, or mark the test with `skip = "requires VFS-absolute paths"` and write expected output manually.

### 1d. Build the spec test runner

The spec test runner is structurally identical to the comparison runner but reads from `tests/fixtures/spec/` and does not support recording mode (spec tests define their expected output manually since our awk/sed/jq are intentionally subset implementations).

**Exit criteria for Phase 1:**
- `cargo test comparison` runs with zero fixture files and passes (no tests = no failures).
- `RECORD_FIXTURES=1 cargo test comparison` records output from a single placeholder fixture.
- The runner produces clear error messages on fixture parsing failures.

## Phase 2 — Shell Language Comparison Fixtures

Write the initial corpus of comparison test cases covering the shell language features that matter most for correctness.

### 2a. Quoting tests

Test single quotes, double quotes, backslash escaping, `$'...'` ANSI-C quoting, mixed quoting, quote removal.

```toml
# tests/fixtures/comparison/quoting/basic_quoting.toml
[[cases]]
name = "single_quotes_literal"
script = "echo 'hello $USER'"
stdout = "hello $USER\n"
exit_code = 0

[[cases]]
name = "double_quotes_expand"
script = 'USER=alice; echo "hello $USER"'
stdout = "hello alice\n"
exit_code = 0

[[cases]]
name = "ansi_c_quoting"
script = "echo $'hello\\tworld'"
stdout = "hello\tworld\n"
exit_code = 0
```

### 2b. Parameter expansion tests

Default values, alternative values, substring extraction, pattern removal, string length, case modification, indirect expansion.

### 2c. Command substitution tests

`$(...)`, backtick form, nested substitutions, substitution in assignments.

### 2d. Arithmetic expansion tests

Basic operators, increment/decrement, ternary, bitwise, variable references, nested arithmetic.

### 2e. Brace expansion tests

Sequence `{1..10}`, string list `{a,b,c}`, nested braces, combined with other expansions.

### 2f. Tilde expansion tests

`~`, `~user` (in sandbox context), tilde in assignments.

### 2g. Word splitting tests

IFS variations, default IFS, custom IFS, empty IFS, quoting preventing splits.

### 2h. Globbing tests

`*`, `?`, `[...]`, no-match behavior, dotfiles, quoting preventing globbing.

Files needed for glob tests are specified in the fixture's `files` field.

### 2i. Redirection tests

`>`, `>>`, `2>`, `&>`, `<`, here-documents (`<<EOF`), here-strings (`<<<`), file descriptor duplication.

### 2j. Pipeline tests

Simple pipes, multi-stage pipes, pipefail, pipe with redirections.

### 2k. Control flow tests

`if`/`elif`/`else`, `for` loops (word list and C-style), `while`/`until`, `case` patterns, `break`/`continue`, nested control flow.

### 2l. Function tests

Function definition (both syntaxes), local variables, return values, recursive functions, function in pipelines.

### 2m. Here-document tests

Quoted delimiter (no expansion), unquoted delimiter (with expansion), `<<-` tab stripping, here-strings.

**Exit criteria for Phase 2:**
- At least 50 comparison test cases across the highest-value categories (expansion, word splitting, quoting). Expand over time.
- All fixtures are recorded from real bash (`RECORD_FIXTURES=1`).
- `cargo test comparison` passes for all cases (or known failures are marked with `skip`).

## Phase 3 — Spec Tests for Command Mini-Languages

Write structured spec tests for `awk`, `grep`, `sed`, and `jq`. These test our implementations' correctness against expected behavior, independent of real bash.

### 3a. `grep` spec tests

- Basic literal match, regex patterns, `-i` (case insensitive), `-v` (invert), `-c` (count), `-l` (files with matches), `-n` (line numbers), `-E` (extended regex), `-F` (fixed strings), `-w` (word match), `-o` (only matching), `-r` (recursive), `-q` (quiet), multiple patterns (`-e`), exit codes (0=match, 1=no match, 2=error).

### 3b. `sed` spec tests

- `s/pattern/replacement/` with flags (`g`, `i`, `p`), address ranges (line numbers, patterns), delete (`d`), print (`p`), append (`a`), insert (`i`), change (`c`), transliterate (`y`), hold/pattern space (`h`, `g`, `x`), multiple expressions (`-e`), in-place (`-i`), regex groups.

### 3c. `awk` spec tests

- Field splitting and `$0`/`$1`..`$NF`, patterns (`BEGIN`/`END`/regex/expression), built-in variables (`NR`, `NF`, `FS`, `OFS`, `RS`, `ORS`), string functions (`length`, `substr`, `index`, `split`, `sub`, `gsub`, `match`, `sprintf`, `tolower`, `toupper`), arithmetic, associative arrays, printf, control flow (`if`/`for`/`while`), `-F` field separator, `-v` variable assignment, multiple rules.

### 3d. `jq` spec tests

- Basic filters (`.`, `.field`, `.[]`), pipe operator, string interpolation, types and values, comparison operators, `if`-`then`-`else`, `try`-`catch`, `map`, `select`, `empty`, `reduce`, `group_by`, `sort_by`, `unique_by`, `keys`, `values`, `has`, `in`, `to_entries`/`from_entries`, `@base64`/`@uri`/`@html`/`@csv`/`@tsv`, `-r` raw output, `-e` exit status, `--arg`, null input (`-n`), slurp (`-s`).

### 3e. Fixture format for spec tests

Spec tests use the same TOML format. For cases needing stdin, use the `stdin` field:

```toml
# tests/fixtures/spec/grep/basic.toml
[[cases]]
name = "literal_match"
script = 'echo -e "apple\nbanana\ncherry" | grep banana'
stdout = "banana\n"
exit_code = 0

[[cases]]
name = "no_match_exit_code"
script = 'echo "hello" | grep xyz'
stdout = ""
exit_code = 1

[[cases]]
name = "read_from_stdin"
script = "read LINE; echo $LINE"
stdin = "hello world"
stdout = "hello world\n"
exit_code = 0
```

For cases needing file input, use the `files` field to seed the VFS.

**Exit criteria for Phase 3:**
- At least 10 test cases per command (40+ total spec tests) covering implemented features. Expand as coverage grows.
- All spec tests pass against rust-bash.
- Tests cover both happy paths and error cases.
- Unimplemented features are documented with `skip` and a reason.

## Phase 4 — CI Integration & Documentation

### 4a. Verify `cargo test` integration

- Ensure all comparison and spec tests run as part of `cargo test`.
- Verify test output is readable — each failure should show the test name, file path, expected vs actual.
- Ensure tests are not flaky (no timing dependencies, no host-dependent paths).

### 4b. Document the testing workflow

Add a section to the guidebook (or update Chapter 9) describing:
- How to add new comparison test cases (just add a TOML entry).
- How to re-record fixtures (`RECORD_FIXTURES=1 cargo test comparison`).
- How to mark known failures with `skip`.
- How spec tests differ from comparison tests.

### 4c. Add a recording CI step (optional, manual)

Document (but don't implement in CI) the workflow for periodically re-recording fixtures to catch regressions against newer bash versions. This is a manual process: a developer runs `RECORD_FIXTURES=1 cargo test comparison` locally, reviews the diffs, and commits updated fixtures.

**Exit criteria for Phase 4:**
- `cargo test` runs all comparison and spec tests.
- Documentation is updated in Chapter 9 or a new section.
- No test flakiness on repeated runs.

## Testing & Rollout

- **Phase 1** is the foundation and the hardest phase — the runner, recording mode, and `datatest-stable` integration must be solid before writing test content.
- **Phase 2** and **Phase 3** can proceed in parallel once Phase 1 is complete.
- **Phase 4** is a lightweight wrap-up.
- Initial target: 50+ comparison tests + 40+ spec tests = 90+ new test cases. Expand over time as M6 features land and M7 commands are added.
- All new tests run within `cargo test` — no separate test commands needed.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `datatest-stable` API incompatibility | Low | Medium | The crate is stable and widely used (Deno, Move). If it doesn't fit, fall back to a simple `#[test]` per file using a macro + `glob` for discovery. |
| Recording mode can't handle all test cases | Medium | Low | Tests needing VFS-absolute paths are marked `skip` for recording and have manually written expected output. Most tests use relative paths or pure-output scripts. |
| TOML round-trip formatting changes | Low | Low | `toml_edit` preserves formatting. Committed to using it for recording mode. |
| Bash version differences across machines | Low | Medium | Document minimum bash version (5.0+). Record bash version in a comment at the top of each fixture file. |
| Flaky tests from non-deterministic output | Medium | Medium | Avoid date/time, PID, random values in test scripts. Use controlled environment with execution limits. |
| Stderr wording varies across bash versions | Medium | Low | Use `stderr_contains` or `stderr_ignore` for fragile stderr checks. |
| Infinite loop in rust-bash hangs test suite | Low | High | Set `ExecutionLimits` in the builder (max 10,000 iterations, 5s wall-clock) for all comparison tests. |
