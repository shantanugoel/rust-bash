# Milestone 4: Filesystem Backends — Implementation Plan

## Problem Statement

rust-bash currently has a single filesystem backend: `InMemoryFs`. While this is the correct default for sandboxed execution, real-world embedding scenarios need more flexibility:

- **Code analysis tools** want to read a real project directory while sandboxing all writes (OverlayFs).
- **Trusted execution** environments want direct real filesystem access with optional path restriction (ReadWriteFs).
- **Complex setups** want to combine multiple backends at different mount points — e.g., in-memory scratch at `/tmp`, overlay on a real project at `/project`, in-memory root elsewhere (MountableFs).

M4 delivers all three backends as specified in Chapter 5 of the guidebook.

## M3 Carryover

M3 is fully complete. All 10 execution limits are enforced with structured `LimitExceeded` errors, counter sharing across subshells works correctly, `NetworkPolicy` and `curl` are implemented and tested. No carryover work needed.

## Key Design Challenges

### 1. Subshell Isolation (`deep_clone`)

The interpreter currently uses `as_any().downcast_ref::<InMemoryFs>()` for deep cloning in three places (subshells, command substitution, exec callbacks). The fallback is `Arc::clone()` (shared, not isolated). This must be generalized so all backends support proper subshell isolation.

**Solution**: Add a `fn deep_clone(&self) -> Arc<dyn VirtualFs>` method to the `VirtualFs` trait. Each backend implements it appropriately:
- `InMemoryFs`: clones entire tree (existing behavior)
- `OverlayFs`: clones the upper (in-memory) layer and whiteout set; lower stays shared (it's read-only)
- `ReadWriteFs`: returns `Arc::clone(self)` — no isolation needed since it's a passthrough to the real FS; subshell writes hit the real FS too
- `MountableFs`: recursively deep-clones each mounted backend

This eliminates all `downcast_ref::<InMemoryFs>()` calls in the interpreter and makes the isolation mechanism backend-agnostic.

### 2. Builder API for Custom Filesystems

The builder currently hardcodes `InMemoryFs::new()` in `build()`. We need a `.fs(Arc<dyn VirtualFs>)` method that lets users provide any backend. When `.fs()` is set, the builder skips creating `InMemoryFs` and uses the provided backend. The `.files()` method still works (writes seed files into whatever backend is provided).

### 3. Real Filesystem Access in OverlayFs/ReadWriteFs

These backends use `std::fs` internally. This is acceptable because:
- The `VirtualFs` trait is the sandbox boundary — **commands** never call `std::fs`
- Backend implementations are trusted code chosen by the embedding application
- The user explicitly opts into real FS access by constructing these backends

## Approach

Four phases, each independently testable. Phase 1 establishes the trait-level foundation (deep_clone + builder). Phases 2–4 implement backends in order of increasing complexity. Each backend gets its own module file under `src/vfs/`.

### Dependencies

**New dev-dependency:**
- `tempfile` (latest) — for creating temporary directories in ReadWriteFs and OverlayFs tests

**No new runtime crates needed.** `std::fs` provides everything for real FS access. `parking_lot` is already available for interior mutability.

## File Organization

```
src/vfs/
```

## Phase 1 — Trait Evolution & Builder Update (Foundation)

> Generalize subshell isolation and allow custom filesystem backends.

### 1a. Add `deep_clone()` to `VirtualFs` trait

Add a new required method to the trait:

```rust
/// Create an independent deep copy for subshell isolation.
///
/// Subshells `( ... )` and command substitutions `$(...)` need an isolated
/// filesystem so their mutations don't leak back to the parent. Each backend
/// decides what "independent copy" means:
/// - InMemoryFs: clones the entire tree
/// - OverlayFs: clones the upper layer and whiteouts; lower is shared
/// - ReadWriteFs: no isolation (returns Arc::clone — writes hit real FS)
/// - MountableFs: recursively deep-clones each mount
fn deep_clone(&self) -> Arc<dyn VirtualFs>;
```

Implement for `InMemoryFs` by wrapping the existing `deep_clone()` method:

```rust
fn deep_clone(&self) -> Arc<dyn VirtualFs> {
    Arc::new(InMemoryFs {
        root: Arc::new(RwLock::new(self.root.read().clone())),
    })
}
```

### 1b. Remove `as_any()` from `VirtualFs` trait

With `deep_clone()` on the trait, `as_any()` is no longer needed. Remove it from:
- The trait definition (`vfs/mod.rs`)
- The `InMemoryFs` implementation (`vfs/memory.rs`)
- All `downcast_ref::<InMemoryFs>()` callsites in the interpreter

### 1c. Update interpreter to use trait-level `deep_clone()`

Replace the three `downcast_ref` patterns in the interpreter with direct trait calls:

**In `expansion.rs:execute_command_substitution()`:**
```rust
// Before:
let cloned_fs: Arc<dyn VirtualFs> =
    if let Some(memfs) = state.fs.as_any().downcast_ref::<InMemoryFs>() {
        Arc::new(memfs.deep_clone())
    } else {
        Arc::clone(&state.fs)
    };

// After:
let cloned_fs = state.fs.deep_clone();
```

Same change in `walker.rs:execute_subshell()` and `walker.rs:make_exec_callback()` (two sites — one for the outer clone, one inside the closure).

### 1d. Add `.fs()` method to `RustBashBuilder`

Add a new field and method to the builder:

```rust
pub struct RustBashBuilder {
    // ... existing fields ...
    fs: Option<Arc<dyn VirtualFs>>,
}
```

```rust
/// Use a custom filesystem backend instead of the default InMemoryFs.
///
/// When set, the builder uses this filesystem directly. The `.files()` method
/// still works — it writes seed files into the provided backend via VirtualFs
/// methods. Note: for `ReadWriteFs`, this means seed files are written to the
/// real filesystem. For `OverlayFs`, seed files go to the in-memory upper layer.
pub fn fs(mut self, fs: Arc<dyn VirtualFs>) -> Self {
    self.fs = Some(fs);
    self
}
```

Update `build()` to use the provided fs or create `InMemoryFs` as default:

```rust
let fs: Arc<dyn VirtualFs> = self.fs.unwrap_or_else(|| Arc::new(InMemoryFs::new()));
```

The rest of `build()` (mkdir_p for cwd, seeding files) works unchanged since it goes through `&dyn VirtualFs`.

### 1e. Add `deep_clone()` for `InMemoryFs` to public API

The existing `InMemoryFs::deep_clone(&self) -> Self` method remains public as a convenience for users who have a concrete `InMemoryFs`. The new trait method `VirtualFs::deep_clone(&self) -> Arc<dyn VirtualFs>` is the generic version.

### 1f. Export new types from `lib.rs`

After all backends are added (Phases 2–4), update `lib.rs` to export:
```rust
pub use vfs::{InMemoryFs, OverlayFs, ReadWriteFs, MountableFs, VirtualFs};
```

Do the `OverlayFs`, `ReadWriteFs`, `MountableFs` exports incrementally as each backend is implemented.

**Exit criteria**: All existing tests pass. `downcast_ref::<InMemoryFs>()` no longer appears in the interpreter. The builder accepts custom filesystems.

## Phase 2 — ReadWriteFs (Simplest Backend)

> Thin `std::fs` wrapper with optional path restriction.

ReadWriteFs is implemented first because it's the simplest new backend and validates the `VirtualFs` trait + `deep_clone()` pattern end-to-end before tackling the more complex backends.

### 2a. Create `src/vfs/readwrite.rs`

```rust
pub struct ReadWriteFs {
    root: Option<PathBuf>,  // Optional chroot-like path restriction
}
```

**Constructor API:**
```rust
impl ReadWriteFs {
    /// Create a ReadWriteFs with unrestricted access to the real filesystem.
    pub fn new() -> Self { ... }

    /// Create a ReadWriteFs restricted to paths under `root`.
    ///
    /// All paths are resolved relative to `root`. Path traversal beyond
    /// `root` (via `..` or symlinks) is rejected with `PermissionDenied`.
    pub fn with_root(root: impl Into<PathBuf>) -> std::io::Result<Self> { ... }
}
```

**Thread safety:** No interior state to lock — all operations delegate to `std::fs` which is inherently thread-safe for independent operations. For interior mutability needs (e.g., none currently), `parking_lot::RwLock` is available.

### 2b. Path restriction logic

When `root` is set, implement a `resolve(&self, path: &Path) -> Result<PathBuf, VfsError>` helper:

1. Join `root` with `path` (stripping any leading `/` from `path` to make it relative)
2. Walk up to find the deepest existing ancestor, canonicalize that ancestor via `std::fs::canonicalize()`
3. Append the remaining (non-existent) path components to the canonicalized ancestor
4. Verify the result starts with `root` — reject with `VfsError::PermissionDenied` if not

**Why not just `canonicalize()` the full path?** `std::fs::canonicalize()` requires the target path to exist. Operations like `write_file` and `mkdir` create new paths that don't exist yet. By canonicalizing the existing ancestor and appending the tail, we correctly handle both existing and not-yet-existing paths.

**TOCTOU note:** Between path resolution and the actual `std::fs` operation, symlinks could theoretically be swapped. This is inherent to real-FS operations and matches the behavior of other chroot-like implementations. Document this in a `# Safety` section on `ReadWriteFs`.

When `root` is `None`, paths are used as-is.

### 2c. Implement all `VirtualFs` methods (21 original + `deep_clone` = 22 total)

Map each method to its `std::fs` equivalent:

| VirtualFs method | std::fs equivalent |
|---|---|
| `read_file` | `std::fs::read` |
| `write_file` | `std::fs::write` |
| `append_file` | `OpenOptions::new().append(true).open().write_all()` |
| `remove_file` | `std::fs::remove_file` |
| `mkdir` | `std::fs::create_dir` |
| `mkdir_p` | `std::fs::create_dir_all` |
| `readdir` | `std::fs::read_dir` |
| `remove_dir` | `std::fs::remove_dir` |
| `remove_dir_all` | `std::fs::remove_dir_all` |
| `exists` | `std::fs::exists` or `Path::exists()` |
| `stat` | `std::fs::metadata` |
| `lstat` | `std::fs::symlink_metadata` |
| `chmod` | `std::os::unix::fs::PermissionsExt::set_mode()` |
| `utimes` | `filetime` crate or `std::fs::File::set_times()` (Rust 1.75+) |
| `symlink` | `std::os::unix::fs::symlink` |
| `hardlink` | `std::fs::hard_link` |
| `readlink` | `std::fs::read_link` |
| `canonicalize` | `std::fs::canonicalize` |
| `copy` | `std::fs::copy` |
| `rename` | `std::fs::rename` |
| `glob` | Walk directory tree + pattern matching (reuse `glob_match` from `crate::interpreter::pattern`) |

**Error mapping:** Map `std::io::Error` to `VfsError`:
- `ErrorKind::NotFound` → `VfsError::NotFound`
- `ErrorKind::AlreadyExists` → `VfsError::AlreadyExists`
- `ErrorKind::PermissionDenied` → `VfsError::PermissionDenied`
- `ErrorKind::DirectoryNotEmpty` → `VfsError::DirectoryNotEmpty` (on stable Rust, check error message or use custom mapping)
- Other → `VfsError::IoError`

**Metadata mapping:** Map `std::fs::Metadata` to `vfs::Metadata`:
- `is_file()` → `NodeType::File`
- `is_dir()` → `NodeType::Directory`
- `is_symlink()` → `NodeType::Symlink`
- `len()` → `size`
- `permissions().mode()` → `mode` (unix-specific)
- `modified()` → `mtime`

### 2d. Implement `deep_clone()`

```rust
fn deep_clone(&self) -> Arc<dyn VirtualFs> {
    // ReadWriteFs is a passthrough — there's no in-memory state to isolate.
    // Subshell writes hit the real filesystem, same as the parent.
    Arc::new(Self { root: self.root.clone() })
}
```

### 2e. Implement glob for ReadWriteFs

Walk the real directory tree using `std::fs::read_dir` recursively, applying `glob_match` from `crate::interpreter::pattern`. Must handle:
- `*`, `?`, `[...]` within path components
- `**` for recursive matching
- Return paths relative to cwd when input pattern is relative

Since the InMemoryFs glob helper (`glob_collect`) is tightly coupled to `FsNode`, write a parallel implementation for real FS walking. Extract `glob_match` into a shared utility if not already accessible.

**Root restriction for glob:** When a root restriction is set, the glob walker must verify that every directory it enters (including symlink targets) remains within the root. Use `resolve()` at each directory level to prevent symlinks from escaping the restricted path.

### 2f. Register module and export

- Add `mod readwrite;` to `vfs/mod.rs`
- Add `pub use readwrite::ReadWriteFs;` to `vfs/mod.rs`
- Add `ReadWriteFs` to `lib.rs` re-exports

### 2g. Tests

Write tests in a new `src/vfs/readwrite_tests.rs` (or a section in `tests.rs`). Use `tempfile` or manual temp directories:

- Basic file CRUD (write, read, append, remove)
- Directory operations (mkdir, mkdir_p, readdir, rmdir)
- Symlink and hardlink operations
- Path restriction: operations within root succeed, operations outside root fail with PermissionDenied
- Path traversal attack: `../../etc/passwd` with root set → PermissionDenied
- Symlink escape: symlink pointing outside root → PermissionDenied on canonicalize
- Glob on real directory tree
- `deep_clone()` returns independent instance
- `stat`/`lstat`/`chmod`/`utimes` on real files

- Write to non-existent file with root restriction (validates `resolve()` for non-existent paths)
- Glob doesn't escape root via symlinks

**Exit criteria**: ReadWriteFs passes all tests. Path restriction is secure. `cargo clippy -- -D warnings` clean.

## Phase 3 — OverlayFs (Copy-on-Write)

> Read from a real directory, write to an in-memory layer. Changes never touch disk.

### 3a. Create `src/vfs/overlay.rs`

```rust
pub struct OverlayFs {
    lower: PathBuf,                              // Real directory (read-only source)
    upper: InMemoryFs,                           // In-memory write layer
    whiteouts: Arc<RwLock<HashSet<PathBuf>>>,    // Tracks deletions
}
```

**Why `Arc<RwLock<...>>` for whiteouts?** The `VirtualFs` trait requires `&self` on all methods, including mutating ones like `remove_file`. Whiteout insertion is a mutation, so it needs interior mutability.

**Constructor:**
```rust
impl OverlayFs {
    /// Create an overlay filesystem with `lower` as the read-only base.
    ///
    /// The lower directory must exist. Returns an error if it doesn't.
    pub fn new(lower: impl Into<PathBuf>) -> std::io::Result<Self> {
        let lower = lower.into();
        if !lower.is_dir() {
            return Err(std::io::Error::new(
                std::io::ErrorKind::NotADirectory,
                format!("{} is not a directory", lower.display()),
            ));
        }
        let lower = lower.canonicalize()?;
        Ok(Self {
            lower,
            upper: InMemoryFs::new(),
            whiteouts: Arc::new(RwLock::new(HashSet::new())),
        })
    }
}
```

### 3b. Resolution order for reads

For every read operation (`read_file`, `stat`, `exists`, `readdir`, etc.):

1. **Check whiteouts** — if the absolute path OR any ancestor of the path is in the whiteout set, return `NotFound`. This is critical: if `/project/src` is whiteout-ed (e.g., `rm -rf /project/src`), then `/project/src/main.rs` must also be considered deleted.
2. **Check upper (InMemoryFs)** — if found, return from upper
3. **Check lower (real FS)** — if found, return from real FS
4. Return `NotFound`

Implement a private whiteout check that walks ancestors:
```rust
fn is_whiteout(&self, path: &Path) -> bool {
    let whiteouts = self.whiteouts.read();
    let mut current = path.to_path_buf();
    loop {
        if whiteouts.contains(&current) {
            return true;
        }
        if !current.pop() {
            return false;
        }
    }
}
```

Implement a resolution helper:
```rust
enum LayerResult<T> {
    Whiteout,
    Upper(T),
    Lower(T),
    NotFound,
}

fn resolve_layer(&self, path: &Path) -> LayerResult<...> { ... }
```

### 3c. Write operations → always to upper

All write operations (`write_file`, `append_file`, `mkdir`, `chmod`, `utimes`, `symlink`, `hardlink`) go directly to the upper `InMemoryFs`. If the file exists only in the lower layer and the operation modifies it (e.g., `append_file`), first copy-up from lower to upper, then apply the mutation.

**`mkdir_p` across layers:** `mkdir_p` must walk path components, checking each against both layers. For each component: if it exists in upper or lower (and is not whiteout-ed), skip. If it doesn't exist in either layer, create it in the upper. This ensures `mkdir_p /project/build/release` works correctly when `/project` only exists in the lower layer — it creates only the missing components (`build`, `release`) in the upper, without needing to duplicate `/project` itself.

**Copy-up pattern:**
```rust
fn copy_up_if_needed(&self, path: &Path) -> Result<(), VfsError> {
    if self.upper.exists(path) {
        return Ok(()); // already in upper
    }
    // Read from lower, write to upper
    let content = self.read_from_lower(path)?;
    self.upper.write_file(path, &content)?;
    // Copy metadata (mode, mtime) from lower to upper
    ...
    Ok(())
}
```

### 3d. Delete operations → whiteout + remove from upper

When deleting a file/directory:
1. If it exists in upper, remove from upper
2. Add the path to the whiteout set (this hides it from the lower layer too)

For `remove_dir_all`, recursively add whiteouts for all children visible from both layers.

### 3e. Directory listings → merged

`readdir` must merge entries from both layers:
1. Start with upper's entries for this directory
2. Add lower's entries that are NOT in the whiteout set and NOT already in upper
3. Return the merged, deduplicated list

### 3f. Rename and copy across layers

`rename(src, dst)`:
- Copy-up `src` if only in lower
- Perform rename in upper
- Add whiteout for original `src` path (to hide it from lower)

`copy(src, dst)`:
- Read from the resolved layer (upper or lower)
- Write to upper

### 3g. Path resolution and canonicalize

`canonicalize` is the highest-complexity method in OverlayFs. Each path component must be resolved step-by-step through both layers:

1. Start from root (`/`)
2. For each component in the path:
   a. Check if the component is a symlink in upper (takes precedence) or lower
   b. If symlink, resolve the target through the overlay (recursively, with depth limit)
   c. If directory, continue to next component
3. Return the fully resolved path

Implement a `resolve_component(&self, parent: &Path, name: &str) -> Result<(PathBuf, NodeType), VfsError>` helper that checks upper → lower for a single path step. Build `canonicalize` on top of this with a symlink depth counter (reuse the existing `MAX_SYMLINK_DEPTH = 40` constant or similar).

This mirrors the step-by-step approach used in `InMemoryFs::canonicalize` (~50 lines) but checks two layers instead of one.

### 3h. Glob across layers

Glob must search both layers and merge results:
1. Glob in upper → collect matches
2. Glob in lower (real FS walking) → collect matches not in whiteout set
3. Merge and deduplicate

### 3i. Implement `deep_clone()`

```rust
fn deep_clone(&self) -> Arc<dyn VirtualFs> {
    // Lower is read-only, safe to share.
    // Upper and whiteouts need independent copies.
    Arc::new(OverlayFs {
        lower: self.lower.clone(),
        upper: self.upper.deep_clone(),  // InMemoryFs::deep_clone() -> Self
        whiteouts: Arc::new(RwLock::new(self.whiteouts.read().clone())),
    })
}
```

### 3j. Lower-layer reading helper

Create a private helper module or functions for reading from the real filesystem. These are self-contained within `overlay.rs` to keep the overlay logic independent from `ReadWriteFs`.

Key helpers needed:
- `read_lower_file(path: &Path) -> Result<Vec<u8>, VfsError>` — maps `lower/relative_path`
- `stat_lower(path: &Path) -> Result<Metadata, VfsError>`
- `readdir_lower(path: &Path) -> Result<Vec<DirEntry>, VfsError>`
- `lower_path(&self, vfs_path: &Path) -> PathBuf` — maps VFS absolute path to real FS path under `lower`

### 3k. Register module and export

- Add `mod overlay;` to `vfs/mod.rs`
- Add `pub use overlay::OverlayFs;` to `vfs/mod.rs`
- Add `OverlayFs` to `lib.rs` re-exports

### 3l. Tests

- **Read-through:** file exists only in lower → readable via overlay
- **Write isolation:** write via overlay → exists in upper, lower unchanged
- **Whiteout:** delete file from lower → not found via overlay, still on disk
- **Copy-up on modify:** append to lower-only file → file now in upper with appended content
- **Merged readdir:** files in both layers appear in listing, whiteouts excluded
- **Rename across layers:** rename lower-only file → new name in upper, old name whiteout
- **Glob merging:** glob finds files from both layers
- **deep_clone isolation:** mutations in cloned overlay don't affect original
- **Non-existent lower:** constructor returns error for missing directory
- **Ancestor whiteout hides descendants:** `rm -rf /project/src` then `cat /project/src/main.rs` fails
- **`mkdir_p` through lower-only directories:** intermediate dirs from lower are recognized without copy-up
- **Hardlink after copy-up:** two hardlinked files in lower; modify one via overlay; the other remains unmodified (hardlink relationship broken, matching real overlayfs behavior)

**Exit criteria**: OverlayFs passes all tests. Lower directory is never modified. `cargo clippy -- -D warnings` clean.

## Phase 4 — MountableFs (Composite)

> Combine multiple backends at different mount points with longest-prefix matching.

### 4a. Create `src/vfs/mountable.rs`

```rust
pub struct MountableFs {
    mounts: Arc<RwLock<BTreeMap<PathBuf, Arc<dyn VirtualFs>>>>,
}
```

Using `BTreeMap` for ordered keys, which enables efficient longest-prefix lookup by iterating in reverse from the queried path.

**Builder-pattern constructor:**
```rust
impl MountableFs {
    pub fn new() -> Self { ... }

    /// Mount a filesystem backend at the given path.
    ///
    /// Paths must be absolute. Mounting at "/" provides the default fallback.
    /// Later mounts at the same path replace earlier ones.
    pub fn mount(self, path: impl Into<PathBuf>, fs: Arc<dyn VirtualFs>) -> Self { ... }
}
```

### 4b. Longest-prefix mount resolution

```rust
/// Find the mount that owns the given path.
///
/// Returns the mount's filesystem and the path relative to the mount point.
fn resolve_mount(&self, path: &Path) -> Result<(Arc<dyn VirtualFs>, PathBuf), VfsError> {
    let mounts = self.mounts.read();
    // Iterate mounts in reverse order (longest paths first due to BTreeMap ordering)
    for (mount_point, fs) in mounts.iter().rev() {
        if path.starts_with(mount_point) {
            let relative = path.strip_prefix(mount_point).unwrap_or(Path::new("/"));
            let resolved = if relative.as_os_str().is_empty() {
                PathBuf::from("/")
            } else {
                PathBuf::from("/").join(relative)
            };
            return Ok((Arc::clone(fs), resolved));
        }
    }
    Err(VfsError::NotFound(path.to_path_buf()))
}
```

**Important**: BTreeMap sorts lexicographically. For paths, this means `/project` comes before `/project/src`, which is the wrong order for longest-prefix. We need to iterate in reverse (`rev()`) or use a custom comparator. Actually, since BTreeMap sorts by key and `/project/src` > `/project` lexicographically, `rev()` iteration gives longest-prefix first. Verify this with tests.

### 4c. Implement all `VirtualFs` methods via delegation

Each method follows the pattern:
```rust
fn read_file(&self, path: &Path) -> Result<Vec<u8>, VfsError> {
    let (fs, relative_path) = self.resolve_mount(path)?;
    fs.read_file(&relative_path)
}
```

### 4d. Cross-mount operations

`copy(src, dst)` and `rename(src, dst)` may span mount boundaries:
- If `src` and `dst` resolve to the same mount → delegate directly
- If they resolve to different mounts → for `copy`: read from src mount, write to dst mount; for `rename`: copy + delete (true rename across mounts is not possible)

`hardlink(src, dst)` across mounts should return an error (hardlinks can't span filesystems, matching real Unix behavior).

### 4e. Directory listings at mount boundaries

When listing a directory that is a mount point's parent, the listing should include the mount point as a directory entry even if it doesn't exist in the parent's filesystem. For example, if `/project` is a mount point but the root InMemoryFs has no `/project` directory, `readdir("/")` should still show `project`.

Implementation:
1. Get entries from the underlying mount for this path
2. Add synthetic directory entries for any child mount points at this level
3. Deduplicate

### 4f. Implement `deep_clone()`

```rust
fn deep_clone(&self) -> Arc<dyn VirtualFs> {
    let mounts = self.mounts.read();
    let cloned_mounts: BTreeMap<PathBuf, Arc<dyn VirtualFs>> = mounts
        .iter()
        .map(|(path, fs)| (path.clone(), fs.deep_clone()))
        .collect();
    Arc::new(MountableFs {
        mounts: Arc::new(RwLock::new(cloned_mounts)),
    })
}
```

### 4g. Glob across mounts

Glob patterns may span multiple mounts. For example, `/*` on a MountableFs with mounts at `/` and `/project` should include entries from both.

Implementation:
1. Determine which mounts could match the pattern prefix
2. Delegate glob to each relevant mount
3. Re-prefix results with the mount point
4. Merge and deduplicate

### 4h. Register module and export

- Add `mod mountable;` to `vfs/mod.rs`
- Add `pub use mountable::MountableFs;` to `vfs/mod.rs`
- Add `MountableFs` to `lib.rs` re-exports

### 4i. Tests

- **Basic delegation:** read/write through mount points
- **Longest-prefix:** `/project/src` mount preferred over `/project` for paths under `/project/src/`
- **Cross-mount copy:** copy from one mount to another
- **Cross-mount rename:** rename across mounts (copy + delete semantics)
- **Directory listing at boundaries:** mount points appear as directories
- **Mount at root:** single mount at `/` works as full delegation
- **Multiple mounts:** complex setup with 3+ backends at different points
- **deep_clone isolation:** mutations in clone don't affect original mounts
- **deep_clone with ReadWriteFs mount:** deep_clone creates a new ReadWriteFs pointing at same path (no isolation for real FS mounts — correct behavior, since ReadWriteFs is a passthrough)
- **Glob across mounts:** pattern spanning mount boundary returns merged results
- **No mount found:** operation on unmounted path returns NotFound (when no root mount exists)
- **`exists()` at mount point itself:** mount points are treated as existing directories
- **Full integration:** create shell with MountableFs via builder, run commands

**Exit criteria**: MountableFs passes all tests. Longest-prefix matching is correct. Cross-mount operations work. `cargo clippy -- -D warnings` clean.

```rust
    .fs(Arc::new(overlay))

    .build()?;
let result = shell.exec("cat /real_file.txt")?; // reads from disk
shell.exec("echo new > /real_file.txt")?;        // writes to memory only
// verify disk file unchanged

// MountableFs integration
let mountable = MountableFs::new()
    .mount("/", Arc::new(InMemoryFs::new()))
    .mount("/project", Arc::new(OverlayFs::new("./myproject")?));
let mut shell = RustBash::builder()
    .fs(Arc::new(mountable))
    .build()?;
```

## Build Order and Dependencies

```
Phase 1 (trait + builder) ─── foundation for all backends
  │
Phase 2 (ReadWriteFs) ──────── simplest backend, validates the pattern
  │
Phase 3 (OverlayFs) ────────── depends on InMemoryFs + real FS reading patterns
  │
Phase 4 (MountableFs) ──────── composes other backends, depends on all prior
  │
Phase 5 (integration + docs) ─ depends on all above
```

**Recommended execution**: Phases 1 → 2 → 3 → 4 → 5 (strictly sequential). Phase 1 is a prerequisite for all others. Phase 4 benefits from having both ReadWriteFs and OverlayFs available for realistic composite setups.

## Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| `std::fs` platform differences (Windows) | Medium | Low | Target Unix first (linux); use `#[cfg(unix)]` for mode/symlink. Cross-platform can be deferred. |
| OverlayFs copy-up complexity | Medium | Medium | Start with file-level copy-up only; directory copy-up can be deferred if not needed by common commands |
| MountableFs cross-mount rename semantics | Low | Low | Use copy+delete; document that rename across mounts is not atomic |
| Glob performance on real FS | Medium | Low | Apply the same `max_glob_results` limit; real FS glob is bounded by the execution limit |
| `deep_clone` on MountableFs is expensive | Low | Low | Only called for subshells; typical usage has 2-3 mounts |
| `utimes` API availability | Low | Low | `std::fs::File::set_times()` is stable since Rust 1.75; we're on a recent edition |
| BTreeMap ordering for longest-prefix | Low | Medium | Validate with explicit tests; the lexicographic order of paths is well-defined |

## Open Design Decisions

1. **Should `OverlayFs::new()` accept `Arc<dyn VirtualFs>` instead of `PathBuf` for the lower layer?** The guidebook specifies `PathBuf` (real directory). Using `Arc<dyn VirtualFs>` would be more general (overlay any backend), but adds complexity and deviates from the spec. **Recommendation: Start with PathBuf as specified. Add a `from_layers(lower, upper)` constructor later if needed.**

2. **Should `ReadWriteFs` restrict to Unix?** The `chmod`, `symlink`, `hardlink` methods use Unix-specific APIs. **Recommendation: Use `#[cfg(unix)]` for these methods, returning `VfsError::IoError("not supported on this platform")` on non-Unix. The rest works cross-platform.**

3. **Should `MountableFs::mount()` take `&self` or consume `self`?** Builder pattern (consuming `self`) is cleaner for construction, but doesn't allow runtime mount/unmount. **Recommendation: Builder pattern for construction. Add `mount_at`/`unmount` methods on `&self` (with interior mutability) only if a runtime use case emerges.**
