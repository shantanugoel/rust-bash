# Milestone 7: Command Coverage and Discoverability ŌĆö Implementation Plan

## Problem Statement

rust-bash has 62 registered commands and 18 builtins but lacks several utilities AI agents commonly encounter (`timeout`, `bc`, `gzip`, `tar`, `file`, `rg`, etc.). No command supports `--help`, there is no `help` builtin, no declarative command metadata, and the default VFS is empty ŌĆö `which ls` returns a hardcoded `/usr/bin/ls` path that doesn't exist on the VFS. AI agents cannot self-discover available commands or their usage.

M7 closes these gaps: add `--help` to every command, implement ~20 missing commands, create a proper default filesystem layout, build command fidelity infrastructure, and ship purpose-built AI agent documentation.

## Prerequisites & Current State

**M1ŌĆōM6 are complete.** The interpreter handles arrays, shopt, process substitution, advanced redirections, parameter transformations, and differential testing.

**Command system architecture** (see `src/commands/mod.rs`):
- `VirtualCommand` trait: `name() -> &str`, `execute(args, ctx) -> CommandResult`
- Commands stored in `HashMap<String, Box<dyn VirtualCommand>>` in `InterpreterState`
- `CommandContext` provides: `fs`, `cwd`, `env`, `stdin`, `limits`, `network_policy`, `exec`
- `CommandResult`: `{ stdout: String, stderr: String, exit_code: i32 }`
- Registration: `register_default_commands()` in `src/commands/mod.rs:614-699`

**Key gaps relevant to M7:**
- **No `--help` handling** in any command. No `CommandMeta` struct.
- **No default VFS layout**: `RustBashBuilder::build()` creates only `/` and the CWD. No `/bin`, `/tmp`, `/home`, `/dev`.
- **No default env vars**: Shell starts with empty environment unless caller sets them.
- **`which` command** (`src/commands/utils.rs:574-713`): Uses hardcoded `REGISTERED_COMMANDS` and `SHELL_BUILTINS` arrays. Returns `/usr/bin/{name}` without checking VFS.
- **String-based pipelines**: `CommandResult.stdout` is `String`, not `Vec<u8>`. Binary data (gzip/tar output) would be corrupted by `String::from_utf8_lossy()` round-trips. This is a **blocker for M7.3** (compression).
- **2,731 existing test cases** across integration, Oils conformance, and differential test suites.

## Phase 1 ŌĆö Default Filesystem Layout and Command Resolution (M7.7)

**Goal**: Make the VFS look like a real Unix system out of the box so AI agents can navigate it naturally and `which` resolves commands correctly.

### Step 1.1 ŌĆö Default directory structure

In `RustBashBuilder::build()` (or a helper it calls), create the standard directories after filesystem construction:

```
/bin/
/usr/bin/
/tmp/             (mode 1777)
/home/user/       (or derive from $HOME if set)
/dev/             (with /dev/null, /dev/zero, /dev/stdin, /dev/stdout, /dev/stderr)
```

Directories are created only if they don't already exist (don't clobber user-seeded files).

### Step 1.2 ŌĆö Command stub files

After `register_default_commands()`, iterate the command HashMap and write a stub file for each command to `/bin/<name>`:

```
#!/bin/bash
# built-in: <name>
```

This makes `ls /bin` list all available commands and `test -x /bin/grep` return true. Stub content is a comment ŌĆö executing the file still goes through normal command dispatch.

### Step 1.3 ŌĆö Default environment variables

Set sensible defaults in `build()` for variables not already provided by the caller:

### Step 1.4 ŌĆö Fix `which` command

Replace hardcoded `REGISTERED_COMMANDS`/`SHELL_BUILTINS` arrays with PATH-based resolution:

1. If the name is a shell builtin ŌåÆ output `{name}: shell built-in command` (check `InterpreterState` or a definitive builtins list).
2. If the name is a user-defined function ŌåÆ output the function name.
3. Split `$PATH` and for each directory, check `ctx.fs.exists("{dir}/{name}")`. Return the first hit.
4. Fall back to checking registered commands map directly (for commands registered but somehow not in `/bin`).
5. Exit 1 if not found.

Remove the `REGISTERED_COMMANDS` and `SHELL_BUILTINS` constant arrays from `utils.rs` ŌĆö they are now redundant since commands are discoverable via VFS.

> **POSIX deviation note**: Real `/usr/bin/which` only searches PATH and does not know about shell builtins ŌĆö it is the `type` builtin that reports builtins. rust-bash's `which` reports builtins because in a sandboxed environment there is no real `/usr/bin/which` binary, and AI agents commonly use `which` to discover available tools. The `type` builtin (already implemented in `builtins.rs`) provides the POSIX-correct behavior.

## Phase 2 ŌĆö Help Infrastructure (M7.1)

**Goal**: Every command responds to `--help` with a usage summary. A `CommandMeta` struct provides declarative metadata.

### Step 2.1 ŌĆö Define `CommandMeta`

Add to `src/commands/mod.rs`:

```rust
pub struct CommandMeta {
    pub name: &'static str,
    pub synopsis: &'static str,        // e.g., "grep [OPTIONS] PATTERN [FILE...]"
    pub description: &'static str,     // One-line summary
    pub options: &'static [(&'static str, &'static str)],  // ("-n", "print line numbers")
    pub supports_help_flag: bool,      // false for echo, true, false, test, [
}
```

Extend `VirtualCommand` trait with a default method:

```rust
pub trait VirtualCommand: Send + Sync {
    fn name(&self) -> &str;
    fn execute(&self, args: &[String], ctx: &CommandContext) -> CommandResult;
    fn meta(&self) -> Option<&CommandMeta> { None }
}
```

**Bash compatibility note**: In real bash, `echo --help` prints the literal string `--help`, and `true --help` / `false --help` / `test --help` / `[ --help` have specific non-help behaviors. The `supports_help_flag: bool` field lets these commands opt out of automatic `--help` interception while still providing metadata for the `help` builtin to use.

### Step 2.2 ŌĆö Dispatch-level `--help` interception

In `dispatch_command()` (in `src/interpreter/walker.rs`), add `--help` checking as the **very first step** ŌĆö before builtin dispatch, before function lookup, before registered command lookup. This ensures `--help` works for builtins too.

Check: if `args[0] == "--help"` AND the command has metadata with `supports_help_flag == true`, format and return the help text as stdout with exit code 0.

For **builtins** (`cd`, `export`, `read`, `printf`, etc.): since builtins are not `VirtualCommand` implementors, maintain a parallel `BUILTIN_META` registry (a `HashMap<&str, CommandMeta>` or a `static` array) that `dispatch_command()` can consult. Note that `printf` exists as both a builtin (priority) and a registered command ŌĆö the `--help` check must work regardless of which dispatch path would normally fire.

Format:
```
Usage: {synopsis}

{description}

Options:
  {flag}    {description}
  ...
```

### Step 2.3 ŌĆö Add `CommandMeta` to all existing commands

Implement `meta()` for all 62 existing commands. Group by file:

- `src/commands/mod.rs`: echo, cat, true, false, pwd, touch, mkdir, ls, test, `[`
- `src/commands/file_ops.rs`: cp, mv, rm, tee, stat, chmod, ln

- `src/commands/exec_cmds.rs`: xargs, find
- `src/commands/sed.rs`: sed
- `src/commands/awk/mod.rs`: awk
- `src/commands/jq_cmd.rs`: jq
- `src/commands/diff_cmd.rs`: diff
- `src/commands/net.rs`: curl

Use `static` `CommandMeta` constants to avoid allocation.

pub fn unknown_option(cmd: &str, flag: &str) -> CommandResult {
    let msg = if flag.starts_with("--") {

    } else {

    CommandResult {

        exit_code: 2,

}

## Phase 5 ŌĆö Compression and Archiving (M7.3)

**Goal**: Implement `gzip`, `gunzip`, `zcat`, and `tar`. This phase first addresses the binary data pipeline blocker, scoped minimally to unblock compression.

### Step 5.1 ŌĆö Binary data audit and minimal fix

Audit the pipeline data path in `src/interpreter/walker.rs` and `src/commands/mod.rs`. The changes span multiple types and functions:

**Types to modify:**

1. **`CommandResult`** (`src/commands/mod.rs`): Add `stdout_bytes: Option<Vec<u8>>`. When a command produces binary output, it sets `stdout_bytes` and leaves `stdout` empty.
2. **`ExecResult`** (`src/interpreter/mod.rs`): Add `stdout_bytes: Option<Vec<u8>>` (mirrors `CommandResult`).
3. **`CommandContext`** (`src/commands/mod.rs`): Add `stdin_bytes: Option<&'a [u8]>`. Commands that handle binary input check this first, falling back to `stdin: &str`.

**Functions to modify:**

4. **`execute_pipeline()`** (`walker.rs:169`): Add a `pipe_data_bytes: Option<Vec<u8>>` alongside `pipe_data: String`. After each pipeline stage, if the command set `stdout_bytes`, propagate the bytes to the next stage's `stdin_bytes`.
5. **`dispatch_command()`** / `execute_simple_command()`**: The `CommandResult ŌåÆ ExecResult` conversion must preserve the `stdout_bytes` field.
6. **`apply_output_redirects()`**: When writing binary output to VFS via redirection (`gzip file > output`), use `stdout_bytes` directly as `&[u8]` rather than converting `stdout` string to bytes.
7. **`exec()` return boundary** (`src/api.rs`): At the public API return, if `stdout_bytes` is set, attempt UTF-8 conversion; if it fails, use lossy conversion. Add an `encoding_hint: Option<String>` field to `ExecResult` for callers that need to know about binary content.

This is a **scoped subset** of M9.8 ŌĆö enough to make compression commands work in pipelines without redesigning the entire `String`-based architecture. Keep the `stdout: String` path as the default for all text commands (they leave `stdout_bytes` as `None`).

## Risk Register Additions

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Binary data scope creep | Medium | Medium | Phase 5 does minimal `stdout_bytes` addition, not full M9.8 redesign. Document boundaries. Explicitly list all functions needing changes (see Step 5.1). |
| `bc` complexity | Medium | Low | Recursive-descent parser with explicit scope: arithmetic, `scale`, comparisons, assignment only. No control flow, functions, or `ibase`/`obase`. Right-associative `^` needs care. |
| `CommandMeta` boilerplate | Low | Low | Use `static` constants. Consider a macro if metadata becomes unwieldy across 77+ commands. |
| `rg` feature parity | Medium | Low | Implement core flags only (`-i`, `-n`, `-r`, `-l`, `-t`, `-g`, `--vimgrep`). Simplified `.gitignore` (root + CWD only). Not full ripgrep. |
| Default env vars breaking tests | Medium | High | Many existing tests may rely on empty env. Use "set only if not present" logic. **Run full test suite immediately after Step 1.3 before any other Phase 1 work.** Audit Oils spec tests for `$HOME`/`$PATH` expectations. |
| `time` user/sys accuracy | Low | Low | Sandbox cannot report real CPU time. Document that `user`/`sys` are always `0.000s`. |
| `sh` mode complexity | Medium | Low | Three distinct modes (`-c`, file, stdin) need different code paths. Reuse `execute_program()`/`execute_subshell()` where possible. |
