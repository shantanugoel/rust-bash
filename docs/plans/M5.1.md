# Milestone 5.1: CLI Binary — Implementation Plan

## Problem Statement

rust-bash is a fully-featured sandboxed bash interpreter (80 commands, 4 filesystem backends, execution limits, network policy) available only as a Rust library crate. To make it usable by people who don't write Rust — and to enable quick shell experimentation from the terminal — we need a standalone CLI binary.

M5.1 delivers a single static binary (`rust-bash`) that supports:
- **`-c 'command'`** — execute a command string and exit
- **Script file argument** — `rust-bash script.sh` reads and executes the file
- **Stdin reading** — `echo 'echo hello' | rust-bash` reads piped input
- **Interactive REPL** — readline-based shell when run with no input and a TTY
- **`--files`** — seed the virtual filesystem from host files/directories
- **`--cwd`** — set the initial working directory
- **`--env`** — set environment variables
- **`--json`** — machine-readable JSON output

Chapter 8 of the guidebook specifies the target interface.

## M1–M4 Carryover

All previous milestones are complete (✅ across M1–M4). No open items need to be addressed.

The existing `examples/shell.rs` provides a development REPL with `--env` and `--files` flags using `rustyline`. Its logic (readline integration, host-directory file loading, prompt formatting, command completion) will be adapted for the production CLI. The example itself will be kept as a minimal library-usage demonstration.

## Key Design Decisions

### 1. Feature-gated CLI dependencies

`clap` and `rustyline` add significant compile time and binary size. Library-only consumers should not pay this cost. Solution: gate them behind a `cli` feature (default-enabled) using optional dependencies.

```toml
[features]
default = ["cli"]
cli = ["dep:clap", "dep:rustyline"]

[[bin]]
name = "rust-bash"
required-features = ["cli"]
```

Library users adding `rust-bash = { ..., default-features = false }` won't pull in CLI dependencies. The `[[bin]]` section with `required-features` ensures the binary isn't built when the feature is off.

### 2. `--files` format

Support two patterns matching the guidebook spec:
- **File mapping**: `--files HOST_PATH:VFS_PATH` — copies a single host file (or directory tree) into the VFS at the specified path.
- **Directory seeding**: `--files HOST_DIR` — recursively copies all files from the host directory into the VFS root.

The first `:` in the value acts as separator. If the value contains no `:`, it's treated as a host directory mapped to VFS root.

### 3. Script file execution

`rust-bash script.sh` reads the host file and executes its contents. This is standard shell behavior and natural to expect, even though the guidebook doesn't list it explicitly.

### 4. Execution mode priority

When multiple input sources are possible, use this precedence (matching standard shell conventions):
1. **`-c 'command'`** — if present, execute the command string and exit
2. **Positional `script.sh`** — if present, read the host file and execute its contents
3. **Stdin is not a TTY** — read all of stdin and execute
4. **Stdin is a TTY** — start interactive REPL

### 5. Exit code behavior

The binary's exit code matches the last command executed, following bash convention. For errors internal to the CLI (bad flags, missing files), exit with code 2 (standard for usage errors). Use `fn main() -> ExitCode` to avoid `std::process::exit()` which bypasses Drop implementations (important for REPL history saving).

### 6. `--json` scope

`--json` only affects **execution output** (results from `-c`, script, or stdin modes). CLI-level errors (invalid flags, missing files, unreadable scripts) always produce plain text on stderr with exit 2, regardless of `--json`. `--json` is not supported in interactive REPL mode — if both `--json` and REPL mode are triggered, the binary prints an error and exits 2.

## Phase 1 — Binary Scaffold & Argument Parsing ✅

> Set up the binary target with all CLI flags defined. Verify `--help` and `--version` work.

### 1a. Update `Cargo.toml`

- Add `[features]` section: `default = ["cli"]`, `cli = ["dep:clap", "dep:rustyline"]`
- Move `rustyline` from `[dev-dependencies]` to `[dependencies]` as `optional = true`
- Add `clap = { version = "...", features = ["derive"], optional = true }` to `[dependencies]`
- Add `assert_cmd` and `predicates` to `[dev-dependencies]`
- Add `[[bin]]` section: `name = "rust-bash"`, `required-features = ["cli"]`
- Verify `[package]` metadata: `description`, `license`, `keywords`

### 1b. Create `src/main.rs` with clap definitions

Define the CLI using clap's derive API:

```rust
use clap::Parser;

/// A sandboxed bash interpreter with a virtual filesystem
#[derive(Parser)]
#[command(name = "rust-bash", version)]
struct Cli {
    /// Execute a command string and exit
    #[arg(short = 'c')]
    command: Option<String>,

    /// Seed VFS from host files/directories (HOST:VFS or HOST_DIR)
    #[arg(long = "files", value_name = "MAPPING")]
    file_mappings: Vec<String>,

    /// Set initial working directory
    #[arg(long, value_name = "DIR")]
    cwd: Option<String>,

    /// Set an environment variable (KEY=VALUE, repeatable)
    #[arg(long, value_name = "KEY=VALUE")]
    env: Vec<String>,

    /// Output results as JSON: {"stdout":"...","stderr":"...","exit_code":N}
    #[arg(long)]
    json: bool,

    /// Script file to execute, followed by optional positional arguments ($1, $2, ...)
    #[arg(trailing_var_arg = true)]
    args: Vec<String>,
}
```

The `args` field captures both the script file (first element) and any positional arguments to pass as `$1`, `$2`, etc. `trailing_var_arg = true` allows arguments after the script name even if they look like flags.

### 1c. Wire up minimal execution

Implement `main()` that:
1. Parses arguments with `Cli::parse()`
2. Builds a `RustBash` instance with `RustBashBuilder::new().build()`
3. If `-c` is present, executes and prints stdout/stderr
4. Returns the command's exit code

Use `fn main() -> ExitCode` (stable since Rust 1.61) to avoid `std::process::exit()`, which skips Drop implementations. This matters for the REPL path where history must be saved. Structure `main()` to compute the exit code, perform all cleanup, then return.

### 1d. Verify build

- `cargo build` succeeds (binary + library)
- `cargo run -- --help` shows formatted usage text
- `cargo run -- --version` shows crate version
- `cargo build --lib --no-default-features` succeeds without pulling in `clap`/`rustyline`

### 1e. Documentation

- Ensure `Cargo.toml` package metadata (`description`, `license`) is accurate

**Exit criteria**: `cargo run -- -c 'echo hello'` prints `hello\n` and exits 0. `cargo run -- --help` shows formatted usage. `cargo build --lib --no-default-features` succeeds.

## Phase 2 — Command Execution & Output Modes ✅

> Implement all four execution modes (-c, script file, stdin, REPL placeholder) and JSON output.

### 2a. Implement `-c` execution

- Execute the command string via `RustBash::exec()`
- Normal mode: print stdout to process stdout, stderr to process stderr
- Exit with the command's exit code

### 2b. Implement `--json` output mode

When `--json` is set, print a single JSON object to stdout:

```json
{"stdout":"hello\n","stderr":"","exit_code":0}
```

Use `serde_json::json!()` (already in dependencies) to construct the output. The process still exits with the command's exit code (the JSON `exit_code` field is a convenience for parsers, not a replacement for the process exit code). This matches standard CLI tool behavior — `rust-bash --json -c 'false' && echo ok` correctly does not print "ok".

**`--json` scope:** `--json` only affects **execution output**. CLI-level errors (bad flags, missing host file in `--files`, unreadable script file) always produce plain text on stderr with exit code 2, regardless of `--json`. This matches how tools like `jq` and `gh` behave.

### 2c. Implement script file execution

When positional arguments are provided (`cli.args` is non-empty):
1. The first element is the script path; remaining elements are positional arguments (`$1`, `$2`, ...)
2. Read the script file from the host filesystem (`std::fs::read_to_string()`)
3. Set `$0` to the script path (via the interpreter's `shell_name` field)
4. Set positional parameters `$1`, `$2`, ... from the remaining arguments
5. Execute via `RustBash::exec()`
6. Output results in normal or JSON mode
7. If the file doesn't exist or can't be read: print error to stderr, exit 2

### 2d. Implement stdin execution

When no `-c` or `script` is given and stdin is not a TTY:
1. Read all of stdin (`std::io::read_to_string()`)
2. Execute via `RustBash::exec()`
3. Output results in normal or JSON mode

Use `std::io::IsTerminal` to detect TTY.

### 2e. Mode dispatch skeleton

Implement the mode priority logic:
```
if cli.command.is_some() → execute_command()
else if !cli.args.is_empty() → execute_script()
else if !stdin.is_terminal() → execute_stdin()
else → start_repl() // placeholder for Phase 4
```

### 2f. Documentation

- Update guidebook Chapter 8: remove **(planned)** from CLI section header
- Ensure `-c`, stdin, and script file examples in Chapter 8 are accurate

**Exit criteria**: `rust-bash -c 'echo hello'` prints `hello\n` and exits 0. `echo 'echo hi' | rust-bash` prints `hi\n`. `rust-bash script.sh arg1` executes the script with `$1=arg1`. `--json -c 'echo x'` outputs valid JSON. CLI errors (bad file path) exit 2 with plain stderr.

## Phase 3 — File Seeding & Environment Configuration ✅

> Support `--files`, `--env`, and `--cwd` flags to configure the sandbox before execution.

### 3a. Implement `--files` parsing

For each `--files` value:
- **Contains `:`**: Split on the first `:` into `(host_path, vfs_path)`.
  - If `host_path` is a file: read its bytes, insert at `vfs_path`
  - If `host_path` is a directory: recursively load all files, prefixed with `vfs_path`
- **No `:`**: Treat the entire value as a host directory, recursively load into VFS root (`/`)

Adapt the `load_host_dir()` function from `examples/shell.rs`.

**Error handling:**
- Host path doesn't exist → `"error: path not found: {path}"`, exit 2
- Not a file or directory → `"error: not a file or directory: {path}"`, exit 2

### 3b. Implement `--env` parsing

For each `--env` value:
- Split on the first `=` into `(key, value)`
- No `=` found → `"error: invalid --env format, expected KEY=VALUE: {val}"`, exit 2

Provide default environment when no `--env` flags are given:
- `HOME=/home`, `USER=user`, `PWD=<cwd or />`

User-provided `--env` values override defaults.

### 3c. Implement `--cwd`

- Pass to `RustBashBuilder::cwd()`
- Default to `/` if not specified

### 3d. Wire all flags into `RustBashBuilder`

```rust
let mut builder = RustBashBuilder::new()
    .files(files)
    .env(env);
if let Some(ref cwd) = cli.cwd {
    builder = builder.cwd(cwd.clone());
}
let mut shell = builder.build()?;
```

### 3e. Documentation

- Update guidebook Chapter 8: ensure `--files` examples match the implemented HOST:VFS syntax
- Update README.md: add a "CLI Usage" section showing file seeding and env examples

**Exit criteria**: `--files /tmp/test.txt:/data.txt -c 'cat /data.txt'` outputs file contents. `--env FOO=bar -c 'echo $FOO'` prints `bar\n`. `--cwd /app -c pwd` prints `/app\n`. Invalid `--files` path exits 2.

## Phase 4 — Interactive REPL ✅

> Start a readline-based interactive shell when no command is given and stdin is a TTY.

### 4a. Adapt REPL from `examples/shell.rs`

Move and adapt the following into `src/main.rs`:
- **`ShellHelper`** struct implementing `Completer`, `Validator`, `Highlighter`, `Hinter`
- **Command completion**: autocomplete from `shell.command_names()` (first token only)
- **Input validation**: multi-line support via `RustBash::is_input_complete()`
- **Prompt**: `rust-bash:{cwd}$ ` with color (green = exit 0, red = non-zero)
- **History**: load/save from `~/.rust_bash_history`
- **REPL loop**: readline → exec → print → update prompt color → repeat

### 4b. Integrate REPL with CLI flags

The REPL respects all sandbox configuration flags:
- `--files` — VFS is pre-seeded before the REPL starts
- `--env` — environment is configured before the REPL starts
- `--cwd` — working directory is set before the REPL starts
- `--json` — **rejected in REPL mode**: if `--json` is set and mode dispatch selects REPL, print `"rust-bash: --json is not supported in interactive mode"` to stderr and exit 2. JSON output requires a well-defined start/end, which conflicts with the REPL's interactive prompt. This can be revisited in a future milestone if there's a real use case (e.g., prompt-free machine-driven REPL).

### 4c. Handle edge cases

- **Ctrl-C**: cancel current input, print `^C`, continue REPL
- **Ctrl-D**: graceful exit — save history, exit with `shell.last_exit_code()` (matching real bash behavior: `false` then Ctrl-D exits 1)
- **`exit` command**: `shell.should_exit()` → break loop, use last exit code
- **Empty input**: skip, re-prompt
- **Exec errors** (parse failures, etc.): print error to stderr, set exit code 1, continue REPL

### 4d. Update `examples/shell.rs`

Update the doc comment at the top of the file:

```rust
//! Minimal interactive REPL demonstrating library-level embedding of rust-bash.
//!
//! For the production CLI binary, run: `cargo run` or `rust-bash` (after install).
//! This example shows how to build a custom REPL using the RustBash API directly.
```

### 4e. Documentation

- Update README.md: document interactive REPL usage, key bindings, history file
- Update guidebook Chapter 8: REPL documentation matches implementation

**Exit criteria**: Running `rust-bash` with a TTY starts the REPL with a colored prompt. Tab-completion works. History persists across sessions. Ctrl-D exits. `--json` in REPL mode prints error and exits 2.

### 4f. Handle `examples/shell.rs` compilation

Since `rustyline` moves from `[dev-dependencies]` to an optional `[dependencies]`, the example needs one of:
- Keep `rustyline` in `[dev-dependencies]` as well (simplest; Cargo handles this correctly — dev-dep takes precedence for test/example targets)
- Or add a `[[example]]` section with `required-features = ["cli"]`

Prefer the first approach for simplicity.

## Phase 5 — Testing ✅

> Comprehensive integration tests for all CLI modes and flag combinations.

### 5a. Create `tests/cli.rs`

Use `assert_cmd` to test the compiled binary end-to-end.

**Execution modes:**
- `-c 'echo hello'` → stdout `"hello\n"`, exit 0
- `-c 'exit 42'` → exit code 42
- `-c 'echo err >&2'` → stderr `"err\n"`
- Script file: write temp file with `echo hello`, run `rust-bash temp.sh`, verify output
- Stdin pipe: pipe `"echo hello"` to stdin, verify output

**JSON output:**
- `--json -c 'echo hello'` → parse JSON, verify `stdout`, `stderr`, `exit_code` fields
- `--json -c 'exit 1'` → process exits 1, JSON has `exit_code: 1`
- `--json -c 'echo err >&2'` → JSON `stderr` field contains `"err\n"`

**File seeding:**
- `--files HOST_FILE:VFS_PATH -c 'cat VFS_PATH'` → outputs file contents
- `--files HOST_DIR -c 'ls /'` → lists seeded files
- `--files HOST_DIR:VFS_PREFIX -c 'ls VFS_PREFIX'` → lists under prefix
- Multiple `--files` flags combine correctly

**Environment & working directory:**
- `--env FOO=bar -c 'echo $FOO'` → `"bar\n"`
- `--env A=1 --env B=2 -c 'echo $A $B'` → `"1 2\n"`
- `--cwd /app -c pwd` → `"/app\n"`
- `--cwd` + `--env` + `--files` combined

**Error cases:**
- Nonexistent script file → exit 2, error message on stderr
- Invalid `--env` format (no `=`) → exit 2, error message on stderr
- Nonexistent `--files` host path → exit 2, error message on stderr
- `-c` with empty string → exit 0 (no-op)

**Flag interactions:**
- `-c` takes priority over positional script argument
- `-c` takes priority over stdin

**Script positional arguments:**
- `rust-bash script.sh arg1 arg2` with script containing `echo $1 $2` → `arg1 arg2\n`
- `rust-bash script.sh` with script containing `echo $0` → outputs the script path

**Additional edge cases:**
- `rust-bash -- -c-looking-script.sh` → treats `-c-looking-script.sh` as a script file (verifies `--` stops flag parsing)
- `--cwd /nonexistent -c pwd` → succeeds (builder auto-creates the directory)
- `--files host.txt:/path -c 'cat /path'` with binary host file → binary content passes through correctly
- `-c $'echo a\necho b'` with multi-line command string → `"a\nb\n"`
- `echo -n '' | rust-bash` → exit 0 (empty script is a no-op)
- `--env HOME=/custom -c 'echo $HOME'` → `/custom` (user overrides defaults)
- `--json` without `-c` and with TTY → exit 2, error on stderr
- `--files /path/with:colon` → handled gracefully (first `:` is separator, so this maps host `/path/with` to VFS `colon`; document this edge case)

- `cargo fmt --check` passes
