# Milestone 6.12: Test Coverage Gap Analysis and Backlog

## Purpose

This document turns the current M6.12 gap analysis into an actionable test backlog for Milestone 6.

It has four goals:

1. Record the current state of bash-comparison, spec, and integration coverage.
2. Classify each M6 phase by implementation status and test status.
3. Separate product gaps from harness gaps and test workarounds.
4. Define the next fixture backlog so unimplemented M6 features are tracked now and start passing as implementation lands.

This is a planning and tracking document. It does not change feature scope from `docs/plans/M6.md` or `docs/guidebook/10-implementation-plan.md`.

This is good coverage for pre-M6 shell behavior and for many M1 compatibility edges. It is not yet a strong representation of the broader M6 plan.

This exceeds the original minimum in M6.12, but it is manual-output command testing, not differential bash-comparison coverage.

## Overall Assessment

### What is working

- The M6.12 fixture infrastructure is real, stable, and running in CI via `cargo test`.
- Recording mode exists and updates fixtures from real `/bin/bash`.
- The comparison suite is already valuable for expansion, quoting, control flow, and redirection regressions.
- The spec suite is broad and catches command-level regressions in mini-languages.

### What is missing

- Most M6-specific features are not represented in bash-comparison fixtures.
- Several implemented M6 features are only covered by handwritten integration tests.
- Some current `skip`s are hiding product gaps or harness gaps that should be tracked more explicitly.
- The current fixture format has no first-class concept of "expected failure until feature is implemented".

### Bottom line

M6.12 is successful as infrastructure, but it is not yet sufficient as "M6 bash-comparison coverage". The current green suite mostly validates pre-M6 behavior while much of the real M6 backlog remains either:

- implemented but not differentially tested, or
- unimplemented and not represented in fixtures at all.

## Current Skips and Workarounds

### Current comparison skips

There are 5 skipped comparison cases:

1. Three ANSI-C quoting cases in `quoting/basic_quoting.toml`
2. One stdin-redirection case skipped due to recorder path asymmetry
3. One stderr-redirection ordering case skipped due to rust-bash behavior mismatch

### How to classify them

| Skip | Type | What to do |
|------|------|------------|
| ANSI-C quoting | product gap | convert to expected-fail tracking |
| absolute-path stdin redirection in recording mode | harness gap | fix recorder staging behavior |
| `>&2 2>/dev/null` ordering mismatch | code gap | fix interpreter semantics, then unskip |

### Rule going forward

`skip` should be reserved for cases that the harness cannot execute meaningfully.

Do **not** use `skip` for known product gaps. Use recorded expected-fail cases instead.

## Harness Gaps

### 1. No expected-fail status

The current fixture schema only has:

- normal passing cases
- `skip`
- limited stderr flexibility
- `expect_error`

This is not enough to track planned-but-unimplemented M6 work cleanly.

### 2. `expect_error` is not true differential coverage

The current runner treats an `exec()` error as success when `expect_error = true`, but it does not compare the recorded bash stderr/exit behavior in that path.

This means parse errors and execution errors are not actually compared against bash semantics even though the fixture format suggests they could be.

### 3. Recorder path asymmetry for absolute VFS paths

Recording mode strips the leading slash from fixture-seeded paths and stages them relative to a temp directory. This creates avoidable fixture asymmetry for cases that reference absolute VFS paths.

### 4. No fixture locking or normalization metadata

just-bash uses locked fixtures and per-test normalization to handle legitimate platform differences while keeping coverage visible. rust-bash currently handles the comparable pressure with `skip`, which throws away signal.

## Required Tracking Model

Replace the current binary "pass or skip" mindset with a three-state model.

### Proposed fixture states

| Status | Meaning | CI behavior |
|--------|---------|-------------|
| `pass` | should match today | mismatch fails |
| `xfail` | known product gap, expected not to match yet | mismatch does not fail, match fails as unexpected pass |
| `skip` | harness or platform blocker | excluded from normal pass/fail counts |

### Required fixture metadata

Add these fields for new backlog cases:

```toml
status = "xfail"
milestone = "M6.3"
feature = "shopt.nullglob"
reason = "not implemented yet"
```

### Required runner behavior

- `pass` + mismatch => fail
- `pass` + match => pass
- `xfail` + mismatch => tracked expected failure
- `xfail` + match => fail as unexpected pass so the case gets promoted
- `skip` => ignored in outcome checks, but still counted in summary

### Summary output requirement

The comparison runner should print a short end-of-run summary:

- passing cases by milestone
- expected-fail cases by milestone
- skipped cases by milestone
- unexpected passes
- unexpected failures

This turns the comparison suite into a milestone-progress tracker instead of a flat green/red result.

## Recommended Execution Order

Use this order for the follow-up implementation work:

1. Add xfail support to the fixture format and runner.
2. Convert current product-gap `skip`s to xfail cases.
3. Fix the recorder path asymmetry so absolute-path file cases can be recorded normally.
4. Add comparison fixtures for already-implemented M6 areas:
   - arrays
   - `PIPESTATUS` / `BASH_REMATCH`
   - `declare` attributes
5. Add mixed pass/xfail coverage for `read`.
6. Add xfail fixture files for the remaining unimplemented M6 phases.
7. Update Chapter 9 and Chapter 10 counts once the corpus lands.

## Acceptance Criteria

This backlog should be considered complete when all of the following are true:

- Every M6 phase from M6.1 through M6.11 is represented in the comparison fixture corpus.
- No implemented M6 feature remains covered only by integration tests when bash-comparison coverage is practical.
- Product gaps are represented as xfail fixtures, not hidden behind `skip`.
- Harness-only skips are minimized and clearly documented.
- The comparison runner reports pass / xfail / skip counts by milestone.
- Chapter 9 and Chapter 10 are updated to reflect current corpus counts and current M6 implementation status.

## Non-Goals

This document does not expand M6.12 into full M7 command-fidelity testing.

The comparison suite should eventually grow in that direction, but the backlog here is intentionally focused on Milestone 6 shell-language completeness and on closing the gap between the M6 plan and the current test corpus.
